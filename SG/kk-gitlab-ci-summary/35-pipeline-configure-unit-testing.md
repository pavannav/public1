# Session 35: Pipeline - Configure Unit Testing

## Uploading Unit Test Reports Using Artifacts

GitLab CI/CD allows you to upload unit test reports using the `artifacts` keyword in your pipeline configuration. This ensures reports are available even if the job fails and helps with debugging and visibility.

### Configuring Artifacts for Test Reports

To upload the Mocha test results, add the following to your job in the `.gitlab-ci.yml` file:

```yaml
job_name:
  # ... other job configuration
  artifacts:
    when: always                    # Upload artifacts even if job fails
    expire_in: 3 days               # Retain artifacts for 3 days
    name: "Mocha test result"       # Friendly name for the artifact
    paths:
      - test-results.xml            # Path to the XML file generated by Mocha tests
```

- `when: always`: Ensures the report uploads regardless of job status.
- `expire_in: 3 days`: Artifacts are automatically deleted after 3 days to manage storage.
- `name`: Provides a descriptive label for download purposes.
- `paths`: Specifies the file(s) to upload (e.g., `test-results.xml` generated by `npm test` or `mocha` commands).

> [!NOTE]
> Always verify the exact path of your test results file. For Node.js applications using Mocha, this is typically `test-results.xml` after running the test command.

> [!WARNING]
> Uploading artifacts consumes pipeline storage quotas. Set appropriate `expire_in` values to avoid unexpected costs.

## Using Reports Keyword for Enhanced Visibility

In addition to basic artifacts, GitLab provides the `reports` option under `artifacts` for specialized report types. This enables advanced features like displaying test results in merge requests and pipeline views.

### Supported Report Types

GitLab supports various report formats for different types of tests:

| Report Type         | Use Case                                      | Format          |
|---------------------|-----------------------------------------------|-----------------|
| Accessibility      | Accessibility testing results                | JSON            |
| API Fuzzing         | API fuzzing security tests                   | JSON            |
| Browser Performance | Browser performance metrics                   | JSON            |
| Code Quality       | Static code analysis results                  | JSON            |
| Container Scanning  | Docker image vulnerability scans             | JSON            |
| Coverage           | Code coverage percentages                     | JSON/Cobertura   |
| DAST               | Dynamic Application Security Testing         | JSON            |
| Dependency Scanning | Dependency vulnerability analysis            | JSON            |
| JUnit              | Unit test results (✅ Our focus)              | XML             |
| License Scanning   | License compliance checking                   | JSON            |
| Load Performance   | Load testing performance data                 | JUnit XML       |
| SAST               | Static Application Security Testing          | JSON            |
| Secret Detection   | Credential exposure detection                 | JSON            |

For Node.js unit testing with Mocha, use the **JUnit** report type since Mocha can generate JUnit XML format.

### Configuring JUnit Reports

Add the `reports` section to your artifacts configuration:

```yaml
job_name:
  # ... other job configuration
  artifacts:
    # ... existing artifact configuration
    reports:
      junit: test-results.xml  # Specify JUnit XML file for unit test results
```

- `junit`: Instructs GitLab to parse the XML file as JUnit test results.
- Enables visual reporting in the GitLab UI.

### Benefits of Using Reports

Compared to basic artifacts, `reports` provides:

✅ **Merge Request Integration**: Test failures appear directly in merge request views, helping reviewers assess code quality.  

✅ **Pipeline Dashboard**: Test statistics (passed/failed tests, success rates, latency) are displayed in pipeline tabs.  

✅ **Visual Analysis**: Structured data tables show test suite details, individual test statuses, and execution times.  

```diff
+ Key Advantage: Immediate visibility of test results in MRs aids quicker reviews and approvals
- Limitation: Limited to supported formats like JUnit XML; custom formats require basic artifacts
```

## Lab Demo: Updating Pipeline for Unit Testing

### Steps to Implement

1. Open your `.gitlab-ci.yml` file in the repository.

2. Locate or create the unit testing job (e.g., a job that runs `npm test`).

3. Add the `artifacts` configuration:

   ```yaml
   unit_tests:
     image: node:latest
     script:
       - npm install
       - npm test  # Generates test-results.xml
     artifacts:
       when: always
       expire_in: 3 days
       name: "Mocha test result"
       paths:
         - test-results.xml
       reports:
         junit: test-results.xml
     only:
       - merge_requests
   ```

4. Commit the changes to your feature branch:

   ```bash
   git add .gitlab-ci.yml
   git commit -m "Add artifacts and JUnit reports for unit testing"
   git push origin feature-branch
   ```

### Viewing Results

- **Pipeline Execution**: The job will run tests and upload artifacts.
  - Success indicator: "Uploaded 1 artifact with build id X"
  - Artifacts tab appears in job details.

- **Browsing Artifacts**:
  1. Go to Pipelines → Select job → Artifacts
  2. Click "Browse" to download `test-results.xml`
  3. Raw XML view for debugging

- **Test Results in Pipeline**:
  - Navigate to Pipelines → Select pipeline → "Test" tab
  - View summary: Tests executed, failed, errors, success rate, average latency
  - Details table: Suite names, test names, statuses, execution times
  - Click "View Details" for expanded XML table format

> [!IMPORTANT]
> JUnit reports enable merge request reviewers to see test statistics without additional navigation, improving code review quality.

## Key Takeaways

- Use `artifacts` with `when: always` to ensure test reports upload even on failures.
- Leverage `reports: junit` for Node.js/Mocha tests to enable GitLab UI integrations.
- Artifacts expire after specified periods; configure based on retention needs.
- JUnit reports provide visual test analytics in merge requests and pipeline views.

> [!TIP]
> Start with basic artifact uploads, then enhance with reports for better team visibility and automated reviews.
