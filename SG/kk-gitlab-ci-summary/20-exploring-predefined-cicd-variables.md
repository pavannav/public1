## GitLab Merge Requests

### Key Concepts 📝

GitLab Merge Requests (MRs) are a fundamental tool for collaborative software development. They enable developers to propose changes from a source branch to a target branch, facilitating code review, discussion, and safe integration.

💡 **Core Purpose**: An MR acts as a formal proposal to merge changes between branches. This creates a structured workflow where:
- Developers create dedicated branches for their work
- Proposed changes are packaged as commits
- The MR serves as the forum for review and feedback
- Approvals can be required before merging
- Changes are ultimately integrated into the main codebase

### Comparison to GitHub Pull Requests

GitLab MRs and GitHub Pull Requests (PRs) serve identical core purposes but differ in terminology and implementation details:

| Aspect | GitLab Merge Request | GitHub Pull Request |
|--------|---------------------|---------------------|
| **Terminology** | Merge Request | Pull Request |
| **Core Function** | Propose changes for merging | Propose changes for merging |
| **Workflow** | Similar branch-based development process | Similar branch-based development process |
| **Review Process** | Discussion, feedback, approvals | Discussion, feedback, approvals |
| **Target Branch** | Usually the main/default branch | Usually the main/markdown branch |

⚠️ **Key Workflow Differences**:
- Both enable creating branching points for isolated development
- Both support committing proposed code changes and documentation
- Both provide platforms for team discussion and receiving feedback
- Both can require approvals before merging
- The instructor notes that while the core concept is the same, implementation details differ (though not elaborated in this session)
