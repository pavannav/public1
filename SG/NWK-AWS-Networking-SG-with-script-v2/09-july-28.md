### VPC Architecture & Design

**Critical Subnet Planning Rules**
- **IP Conflict Prevention**: Network segmentation between VPCs (10.1.1.0/24 vs 10.1.2.0/24)
- **Availability Zone Redundancy**: Distribute workloads across multiple AZs for high availability
- **Workload Isolation**: Separate subnets for different application tiers (web/application/database)

**Default VPC vs Custom VPC**
- Production deployments ALWAYS use custom VPC configurations
- Default VPCs excluded from enterprise network planning
- Client requirements drive subnet allocation and resource placement

**Subnet Classification Table**

| Subnet Type | IP Range | Purpose | AZ Distribution |
|-------------|----------|---------|-----------------|
| Public (Bastion) | 10.1.1.32/28 | Internet-accessible jump hosts | 3 AZs (1A, 1B, 1C) |
| Private Application A | 10.1.1.0/28 | Internal workload tier | 3 AZs specific IPs |
| Private Application B | 10.1.1.16/28 | Cross-VPC accessible resources | 3 AZs specific IPs |
| Private Database | 10.1.1.24/28 (example) | Data layer isolation | Distributed AZs |

### Security Group Configuration

**Bastion Host Security**
- **Inbound Rules**: SSH (TCP 22) from `0.0.0.0/0` ⚠️ (Controlled via private key only)
- **Outbound Rules**: All protocols allowed (required for internal access)
- **Key Principle**: Private keys stored in memory only - physical copies compromise infrastructure

**Private Subnet Access Control**
- **Source-Based Filtering**: Allow SSH only from bastion host security group
- **Stateful Behavior**: Automatic return traffic allowance via connection tracking
- **Multi-Interface Support**: Security groups apply per network interface

```diff
+ Best Practice: Security group references = Network-level control
- Anti-Pattern: Static IP whitelisting = Maintenance nightmare
+ Expert Tip: Use security group IDs for dynamic membership
```

### VPC Peering Fundamentals

**Peering Requirements**
- Same AWS account = No special permissions
- Cross-account = Account ID and region specification
- Requester/Acceptor workflow for approval

**Network Overlap Detection**
- AWS automatically prevents conflicting CIDR blocks
- Theoretical knowledge mandatory for PC delivery
- Client rejection risk for misconfigurations

**Routing Implications**
- Peering connections = Private communication routes
- IGW bypassed for internal traffic
- Security group scope limited to local VPC

### Route Table Management

**Main Route Table vs. Custom Tables**
- Default association = All subnets (unless explicitly detached)
- Custom tables = Explicit subnet associations
- Traffic routing = Based on most specific route match

**Route Propagation**
- Static routes require manual configuration
- Dynamic propagation = VPN gateways inject routes
- BGP integration for enterprise connectivity

**Gateway Types Table**

| Gateway Type | Purpose | Route Propagation | Scope |
|--------------|---------|-------------------|-------|
| Internet Gateway | Public internet access | No propagation | Regional |
| Virtual Private Gateway | External VPN termination | Route injection | VPC-level |
| Transit Gateway | Multi-VPC routing hub | Full propagation support | Regional |

### SSH Key Access Methods

**Method Categories**
- **Advanced Methods** (Forwarding Agent): Production-ready, secure key management
- **Basic Methods** (File Transfer): Learning tools, not production-ready

**PowerShell/Windows Methods**
```bash
# Start SSH agent
Get-Service ssh-agent | Set-Service -StartupType Automatic
Start-Service ssh-agent

# Add private key to memory
ssh-add path/to/private_key.pem

# Forward key for bastion access
ssh -A ec2-user@bastion-public-ip
```

**Linux Methods**
```bash
# Agent management
eval "$(ssh-agent -s)"
ssh-add ~/Downloads/private_key.pem

# Bastion connection with forwarding
ssh -A -i key.pem ec2-user@bastion-ip
```

**Bastion-to-Private Access**
```bash
# From bastion host (with forwarded keys)
ssh ec2-user@private-ip
```

> [!IMPORTANT]
> Always validate fingerprint authenticity on first connection. Compromised bastions are critical security incidents requiring immediate quarantine.

## Summary

### Key Takeaways
```diff
+ VPC peering establishes private connectivity between VPCs without IGW routing
+ Bastion hosts provide secure jump-box access to private subnets via security group controls
+ Route tables control traffic flow and must be customized for peering scenarios
+ Security groups operate within VPC boundaries and don't span peering connections
+ SSH forwarding agents provide secure, memory-based key management without physical storage
- Network overlaps prevent VPC peering and can block project progression
- Default VPC usage is unacceptable for production client deployments
- Static IP whitelisting creates maintenance burdens in dynamic environments
```

### Expert Insight

**Real-world Application**
VPC peering is fundamental for multi-tier application architectures where web servers, application servers, and databases need internal communication without exposing services to the internet. This pattern scales to complex hub-and-spoke topologies using Transit Gateway for enterprise landing zones.

**Expert Path**
- Master subnet calculation and CIDR planning (critical for AWS Solution Architect certifications)
- Understand AWS service communication patterns (Lambda to RDS via VPC endpoints vs. peering)
- Practice with CloudFormation/Terraform for repeatable network infrastructure
- Learn AWS Direct Connect patterns for reliable cross-region connectivity

**Common Pitfalls**
- **❌ Route Table Confusion**: Mixing main and custom route tables without understanding attachment behavior
- **❌ Security Group Scope**: Assuming security groups work across peering connections
- **❌ IGW Dependencies**: Forgetting that peered traffic bypasses internet gateways
- **❌ Key Management**: Storing private keys on bastion hosts instead of using forwarding agents
- **⚠️ CIDR Conflicts**: Overlapping networks prevent peering and require architecture redesign
- **❌ Single AZ Deployment**: Neglecting availability zone distribution for redundancy

**Lesser Known Facts**
- Peering connections don't support transitive routing (A→B and B→C doesn't enable A→C)
- Security groups don't propagate across peering connections (must create identical rules in target VPC)
- DNS resolution across peering requires enabling DNS hostnames in both VPCs
- ICMP traffic doesn't automatically flow through peering (explicit security group rules required)
- Bastion hosts should be in auto-scaling groups with session manager for production environments
