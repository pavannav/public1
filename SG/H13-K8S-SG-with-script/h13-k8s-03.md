## Overview

This session introduces the fundamentals of Kubernetes, focusing on its role as a container orchestration platform. It explains how Kubernetes acts as a server program that manages applications encapsulated in containers, abstracting the underlying container technology like Docker. The session covers the evolution from manual CLI interactions to declarative YAML-based configurations, emphasizing the importance of vocabulary and concepts in the Kubernetes ecosystem. Key topics include the API server, Minikube for local setup, pod management via CLI and YAML, and an introduction to YAML syntax for defining Kubernetes resources. The goal is to build a basic understanding of how users declare requirements to Kubernetes, which then handles container deployment, networking, and scaling automatically.

### Kubernetes Architecture Overview

- Kubernetes is a program running as a master/server that listens for user requirements via its API server (kube API).
- The API server processes requests to deploy applications inside containers, abstracting complexities like container management.
- Users interact with Kubernetes as a server, which coordinates with underlying container engines to launch, run, and manage containers.
- Kubernetes wraps containers in units called pods for management, security, and networking isolation.
- The master program (including the API server) runs on an operating system, often virtualized (e.g., via Minikube) for ease of setup.

```diff
+ Core Component: API Server - Acts as the primary interface for user requests
- Myth: Kubernetes directly launches containers (it uses cubelet to interface with engines like Docker)
! Key Identity: Kubernetes is a declarative system where users "tell" it what they want
```

### Minikube and Cluster Setup

- Minikube is a tool that automates setup of a local Kubernetes cluster (single-node) by launching a virtual machine (e.g., using VirtualBox or Hyper-V) and installing Kubernetes programs.
- It installs the kube API server and other components like etcd, and by default uses Docker as the container runtime.
- Commands for Minikube management:
  - `minikube start`: Launches the cluster.
  - `minikube status`: Checks cluster status.
  - `minikube stop`: Powers down the cluster cleanly.
- Requirements: At least 2GB free RAM; virtualization enabled in BIOS (VT-x/AMD-V on Intel/AMD CPUs).
- Troubleshooting:
  - **Virtualization Disabled Error**: Enable VT-x in BIOS and restart.
  - **Hyper-V Conflict**: Disable Hyper-V in Windows Features if using VirtualBox.
- After setup, confirm with `kubectl cluster-info` and `kubectl get nodes`.

⚠️ Importance: Minikube simplifies local development but isn't for production—enterprise setups use multi-node clusters.

### Interacting with Kubernetes: CLI vs. YAML

- **CLI (Command Line Interface)**: Direct commands like `kubectl run` for quick actions; limited customization.
- **YAML Declarative Code**: Preferred for complex, repeatable configurations; allows full customization and automation.
- Both methods send requests to the kube API server, but YAML enables advanced features like multi-container pods, metadata, and scalability.

```diff
+ CLI Advantage: Fast for simple tasks (e.g., `kubectl create pod`)
- CLI Limitation: Not flexible for automation or integration
! YAML Power: Supports all Kubernetes features, integrates with CI/CD pipelines
```

| Method | Pros | Cons |
|--------|------|------|
| CLI | Quick, interactive | Limited options, hard to version |
| YAML | Declarative, reusable | Requires syntax knowledge |

📝 Production Tip: YAML files are version-controlled in Git for deployments.

### Container Engines and Kubernetes

- Kubernetes doesn't directly launch containers; it uses programs like kubelet to interface with container engines (e.g., Docker, containerd).
- Docker is common by default in Minikube; Kubernetes sends requests to Docker to pull images and create containers.
- Images encapsulate apps and OSes; users specify image names (e.g., from Docker Hub), and Kubernetes delegates container creation.

```diff
+ Abstraction Benefit: No need to learn engine-specific commands
- Underlying Dependency: Kubernetes requires a container runtime
! Proof: Running `docker ps` in Minikube VM shows pods as containers launched by Kubernetes via Docker
```

> [!IMPORTANT]
> Kubernetes manages the "what" (resources like pods), while container engines handle the "how" (runtime execution).

### Understanding Pods

- Pods are Kubernetes' smallest deployable unit, wrapping one or more containers for shared resources and management.
- Benefits: Isolation, networking, lifecycle management; Kubernetes deletes containers to kill pods, re-launch to restart.
- Resources (e.g., pods) are queried with `kubectl get pods` (basic) or described with `kubectl describe pods`.
- Behind the scenes: Pods run as containers in the container engine, identifiable via `docker ps`.

```diff
+ Why Pods? Enable multi-container apps, shared volumes, and coordinated restarts
- Single-Container Focus for Beginners: Most examples are single-container pods
! Container Naming: Can be customized in YAML, defaults to pod name
```

### Introduction to YAML

- YAML (Yet Another Markup Language) is used for declarative configurations, formatting key-value pairs for data exchange.
- Format: `key: "value"` (case-sensitive, keys like kind, version; strings in quotes optional but recommended).
- Arrays/Lists: Denoted by hyphens (e.g., `containers: - name: "app1"`).
- Indentation: Uses spaces (not tabs) for structure; 2-4 spaces standard, but consistent indentation required.

```diff
+ Declarative Simplicity: Unlike JSON, human-readable for configs
- Strict Indentation: Errors cause failures
! Kubernetes Standard: All resource definitions use YAML
```

Basic YAML Structure Example:
```yaml
key: "value"
array:
  - item1
  - item2
```

> [!NOTE]
> YAML enables sending structured requests to kube API without scripting.

### Writing YAML for Pods

- Essential YAML keys for pods:
  - `apiVersion`: API version (e.g., "v1" for pods).
  - `kind`: Resource type (e.g., "Pod").
  - `metadata`: Metadata like `name`.
  - `spec`: Specifications (e.g., `containers` array).
- Containers defined as list under `spec.containers`, each with `name` and `image`.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: "my-port"
spec:
  containers:
    - name: "my-c1"
      image: "nginxdemos/hello:latest"
```

✨ Customization: YAML allows multiple containers per pod, labels, and more specs not available in CLI.

### Creating and Deleting Resources

- Apply YAML with `kubectl apply -f file.yml` or `kubectl create -f file.yml`.
- Delete with `kubectl delete pods pod-name` (specific) or `kubectl delete all` (destructive, deletes all resources).
- Naming: Use lowercase, no spaces; extensions like `.yml`.

```diff
+ Apply Command: Declarative way to create/update resources
- Delete All Caution: Destroys everything; use sparingly
! File Path: Ensure correct path; set `kubectl` in PATH for global usage
```

### Quick Reference

- **Minikube Commands**:
  - `minikube start`: Start local cluster.
  - `minikube status`: Check cluster/app health.
  - `minikube stop`: Stop cluster.

- **kubectl Basics**:
  - `kubectl get pods`: List pods.
  - `kubectl describe pod <name>`: Detailed pod info.
  - `kubectl apply -f <file.yml>`: Apply YAML manifest.

- **YAML Pod Template**:
  ```yaml
  apiVersion: v1
  kind: Pod
  metadata:
    name: my-pod
  spec:
    containers:
      - name: my-container
        image: nginx:latest
  ```

### Expert Insight

**Real-world Application**: In production, use YAML for infrastructure as code (IaC)—commit manifests to Git and apply via CI/CD to ensure consistent, repeatable deployments across environments.

**Expert Path**: Master YAML syntax variations (e.g., maps, lists), then advance to deployments and services for scaling; practice with multi-container pods for microservices architectures.

**Common Pitfalls**: 
- **Indentation Errors**: Always validate YAML with tools like `yamllint` before applying.
- **Path Issues**: Set `kubectl` in system PATH to avoid "command not found."
- **Overusing `delete all`**: Leads to data loss; target specific resources.
- **Ignoring Specs**: Missing `spec.containers` causes pod creation failures.
- **Misconfigured Images**: Ensure images exist in registries like Docker Hub; slow pulls indicate network issues.

**Lesser-Known Facts**: 
- Kubernetes originally used "kind" for resource types to maintain backward compatibility; early versions focused on Borg-inspired abstractions.
- Pods share network/IP within resources but isolate containers; this enables sidecar patterns (e.g., logging/proxies in multi-container pods).
