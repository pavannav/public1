## Overview
This session revisits the multi-tier application setup from the previous session, where WordPress (web app) and MySQL (database) run in separate pods with interconnectivity. We shift focus to a critical security challenge: managing sensitive data like database passwords. Traditional approaches using environment variables expose passwords in clear text via commands like `kubectl describe pod`, posing risks in shared environments or code repositories. We introduce Kubernetes Secrets as a dedicated resource for storing sensitive key-value pairs, promoting separation of concerns and better security practices without full encryption.

> [!IMPORTANT]
> Secrets enable secure reference to credentials without embedding them directly in application code or deployment YAMLs.

## Key Concepts / Deep Dive

### Security Issues with Passwords
Environment variables pass MySQL root passwords, but `kubectl describe pod` reveals them in clear text—a risk for shoulder surfing or shared screens. Command history (`history`) or YAML files committed to Git/GitHub expose passwords permanently. This violates security best practices, especially in team environments where managers hold credentials while developers write code.

Common Pitfalls:
- **Clear Text Exposure**: Describing pods or running `env` outputs plain credentials.
- **Code Repository Risks**: Accidentally pushing sensitive YAMLs.

### Understanding Kubernetes Secrets
Secrets are a Kubernetes object (like pods/services) for storing sensitive data as key-value pairs. They provide **encoding** (not encryption) to obfuscate data—base64 transforms clear text (e.g., "red" → "cmVk"), preventing quick spotting but allowing easy decoding if accessed.

> [!NOTE]
> Secrets are managed separately; they don't "lock" data but facilitate secure references, akin to vaults in tools like Ansible or HashiCorp Vault.

Purpose: Enable teams to share code without exposing secrets. Managers create secrets; developers reference them. This prevents:
1. Password visibility in `describe` or history.
2. Accidental GitHub uploads (code uses refs, not values).

Secrets flow:
1. Create secret resource with encoded data.
2. Reference secret keys in pod environments (e.g., env vars pull from secret).

```diff
+ Manager Creates Secret (Encoded Values) → Developers Reference in Code → Pod Reads at Runtime
```

### Creating Secrets
Use YAML for declarative management—standard for Kubernetes resources.

**YAML Structure:**
```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret  # Unique name for referencing
type: Opaque      # Generic type for key-value pairs
data:
  p: cmVk         # Key: Base64-encoded value ("red")
```

- `name`: Reference identifier.
- `data`: Key-value pairs (values must be base64-encoded).
- Apply via `kubectl apply -f secret.yaml` or `kubectl create -f secret.yaml`.

**Encoding/Decoding:**
Base64 tools (online or CLI):
- Encode: `echo -n 'red' | base64` → `cmVk`
- Decode: `echo 'cmVk' | base64 -d` → `red`

> [!WARNING]
> Commit secret YAMLs cautiously—though values are encoded, avoid sharing; use RBAC (future topic) to restrict access.

### Using Secrets in Pods
Instead of direct env values, reference secrets to hide passwords.

**Updated Pod YAML:**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-db
  labels:
    app: my-db
spec:
  containers:
  - name: my-db
    image: mysql:5.7
    env:
    - name: MYSQL_ROOT_PASSWORD
      valueFrom:              # Pull value from secret
        secretKeyRef:
          name: my-secret
          key: p
    - name: MYSQL_USER
      value: vml_user         # Static or from secret
```

- `valueFrom.secretKeyRef`: References secret name and key.
- Pod pulls encoded value and decodes internally.

Run: `kubectl create -f pod.yaml`
Check: `kubectl describe pod my-db` shows `valueFrom.secretKeyRef` (not password).

### Lab Demo: Implementing Secrets in MySQL Pod
1. **Prepare Files:**
   - Secret YAML (`secret.yaml`)

2. **Create Secret:**
   ```bash

   ```
   Verify: `kubectl get secret my-secret`

3. **Launch Pod:**
   ```bash

   ```

4. **Verify:**
   - `kubectl logs my-db` (check for success).
   - `kubectl describe pod my-db` (env hides password).
   - Dev notebook or describe shows refs, not values.

- Successful launch; password obscured.
- WordPress connection works via secret-powered MySQL.

### Creating Secrets via Command Line
For quick tests:
```bash
kubectl create secret generic my-secret --from-literal=p=red
```

- `--from-literal`: Key=value pair.
- Suitable for small data; YAML preferred for versioning.

More Options:
- `--from-file`: Load from file (e.g., SSH keys).
- Types: Opaque (generic), TLS (certificates).

## Summary

### Key Takeaways
```diff
+ Store sensitive data in Kubernetes Secrets for obfuscation and secure references.
- Avoid clear-text passwords in env vars, commands, or code to prevent exposure.
! Enable team separation—managers handle secrets, developers reference them.
+ Use base64 encoding for storage; secrets are not encrypted.
- Secrets alone don't guarantee security; combine with RBAC.
```

### Quick Reference
- **Create Secret (YAML):** `kubectl apply -f secret.yaml`
- **Create Secret (CLI):** `kubectl create secret generic <name> --from-literal=<key>=<value>`
- **Check Secret Info:** `kubectl describe secret <name>` or `kubectl get secret <name> -o yaml`
- **Encode Value:** `echo -n '<value>' | base64`
- **Decode Value:** `echo '<b64>' | base64 -d`
- **Pod Env Reference:** 
  ```yaml
  env:
  - name: ENV_VAR
    valueFrom:
      secretKeyRef:
        name: secret-name
        key: key-name
  ```

### Expert Insight

**Real-world Application**: In production, secrets manage database credentials, API keys, and certificates across microservices—e.g., WordPress/MySQL setups in Kubernetes via Helm charts integrate secrets for secure deployments.

**Expert Path**: Advance to RBAC for access control (role-based access), TLS secrets for ingress encryption, and config maps for non-sensitive configs. Study Kubernetes Certificate Management for TLS.

**Common Pitfalls**:
- Forgetting base64 encoding (causes apply errors).
- Committing secret YAMLs to repos—secure with private repos/sealed-secrets.
- Mistaking encoding for encryption—secrets decode easily; use external vaults for full security.
- Over-relying on secrets for everything—non-sensitive data belongs in ConfigMaps.

**Lesser-Known Facts**: Secrets support binary data (e.g., SSH private keys or certs); they mount as files in volumes for apps needing cert files. Empty values efficient (base64 encodes to empty string). Persistent but reloadable without pod restart if referenced correctly. Avoid modifying live secrets—use updates via `kubectl replace` or apply. 🚀
