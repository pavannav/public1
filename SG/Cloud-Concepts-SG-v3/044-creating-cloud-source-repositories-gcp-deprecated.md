## Overview
This session covers Google Cloud Source Repositories (CSR), a Git-based, private hosting service for source code. CSR provides secure hosting for private repositories with integration capabilities for existing GitHub or Bitbucket repositories. The session demonstrates how to create repositories, configure authentication methods, and set up mirroring for external repositories. Note that Cloud Source Repositories has been deprecated as of March 2023 in favor of Cloud Build repositories and other Google Cloud offerings.

## Cloud Source Repositories Overview

### What are Cloud Source Repositories?
Cloud Source Repositories (CSR) are Git-based, private source code repositories hosted on Google Cloud Platform. They function identically to GitHub repositories but are hosted on Google's infrastructure.

**Key Characteristics:**
- Private repositories only
- Full Git support (clone, pull, push, etc.)
- Integration with Google Cloud services
- No public repository option

### Use Cases
- Centralized code storage for Google Cloud projects
- Mirroring existing GitHub or Bitbucket repositories
- Secure CI/CD pipeline integration
- Team collaboration on private codebases

## Repository Creation Options

### Two Main Creation Methods:

1. **Create New Repository**
   - Start from empty repository
   - Set up new Git history
   - Full read/write access from start

2. **Connect External Repository**
   - Mirror existing GitHub or Bitbucket repositories
   - Sync commits automatically
   - Read-only mirror functionality

### Supported External Services:
- GitHub.com repositories
- Bitbucket Cloud repositories
- **Not supported**: Self-hosted GitHub/Bitbucket instances

## Authentication Methods

Three primary authentication mechanisms support different access patterns:

### 1. SSH Key Authentication
- Most secure method
- Requires SSH key pair generation
- Platform agnostic (works on Linux, Windows, macOS)

### 2. Cloud SDK Authentication
- Uses `gcloud` command-line tool
- Automatic credential management
- Integrated with Google Cloud identity

### 3. Manual Credentials Authentication
- Browser-based credential generation
- Temporary access tokens
- Works across all repositories for authenticated user

> **Note:** Each authentication method generates different clone URLs, requiring separate repository clones for testing different methods.

## SSH Key Authentication Setup

### Key Generation Commands:
```bash
# Generate RSA key pair
ssh-keygen -t rsa -C "username@domain.com"

# Follow prompts (use default locations or specify custom path)
```

### CSR SSH Key Registration:
1. Navigate to Cloud Source Repositories console
2. Access repository settings (three-dot menu)
3. Select "Manage SSH Keys"
4. Click "Register SSH Key"
5. Provide key name and paste public key content

### Clone with SSH:
```bash
git clone ssh://[USER]@[PROJECT-ID]@source.developers.google.com:2022/p/[PROJECT-ID]/r/[REPO-NAME]
```

## Manual Credentials Authentication

### Credential Generation Process:
1. In Cloud Source Repositories console, go to "Clone" section
2. Click "Manually generate and store Git credentials"
3. Select associated email account
4. Copy provided credential configuration

### Linux/Mac Configuration:
```bash
# Configuration generated by Google Cloud Console
[credential "https://source.developers.google.com"]
    helper = store
```

### Windows Configuration:
Uses different credential helper format with forward slashes for paths.

### Clone with Manual Credentials:
```bash
git clone https://source.developers.google.com/p/[PROJECT-ID]/r/[REPO-NAME]
```

> **Security Warning:** Manual credentials provide access to ALL repositories in the project. Share these credentials carefully and treat them as sensitive information.

## Mirroring External Repositories

### Supported Platforms:
- GitHub Cloud repositories
- Bitbucket Cloud repositories

### Mirroring Process:

1. **Authentication Setup:**
   - Provide GitHub/Bitbucket credentials
   - Grant Google Cloud access permissions
   - Select specific repositories (not entire accounts)

2. **Permission Levels:**
   - **Reader:** View-only access with sync capabilities
   - **Writer:** Reader permissions plus push rights
   - **Administrator:** Full repository management access

3. **Repository Naming Convention:**
   - Automatically prefixed with platform name
   - Format: `[PLATFORM]-[USERNAME]/[REPO-NAME]`
   - Example: `bitbucket-username/repository-name`

### Important Limitations:

> **Important:** Mirrored repositories are read-only in Cloud Source Repositories. All commits must be made to the original GitHub/Bitbucket repository - changes sync automatically to Google Cloud.

### Manual Sync Option:
Access repository settings → "Sync from [Platform]" to force immediate synchronization.

## Cloud Source Repositories Security Features

### Sensitive File Prevention:
CSR includes built-in security scans that warn against committing sensitive files:

- JSON credential files (`.json`)
- PAM (Pluggable Authentication Module) files
- Other potentially sensitive configuration files

### Warning System:
- Pre-commit warnings (not blocks)
- Security advisories during push operations
- User discretion for proceeding with commits

## Pub/Sub Integration for Notifications

### Event Notification Setup:

1. **Create Pub/Sub Topic:**
   ```bash
   # Not shown in transcript, typical command structure
   gcloud pubsub topics create REPO-EVENTS
   ```

2. **Associate Topic with Project:**
   ```bash
   gcloud source project-configs update --add-topic=projects/[PROJECT-ID]/topics/[TOPIC-NAME]
   ```

### Supported Events:
- Repository creation
- Repository deletion
- Code changes/pushes
- Access permission modifications

### Message Consumption:
- Pull notifications from Pub/Sub subscriptions
- Integrate with Cloud Functions for automation
- Trigger workflows based on repository activity

## Repository Management and Permissions

### Repository-Level Permissions:

**Available Access Levels:**
| Permission Level | Capabilities |
|------------------|--------------|
| Reader | View repository contents, clone, pull |
| Writer | Reader permissions + commit, push |
| Administrator | Writer permissions + delete repository, manage permissions |

### Management Options:
- Delete repositories (requires confirmation)
- Modify access permissions
- View repository settings and configuration
- Access audit logs and activity history

## Lab Demo: Creating and Using Cloud Source Repositories

### Demo Scenario: New Repository Creation with SSH Authentication

**Step 1: Create Repository**
1. Navigate to Cloud Source Repositories console
2. Click "Create Repository"
3. Choose "Create new repository"
4. Provide repository name (e.g., "my-new-repo")
5. Select target GCP project
6. Click "Create"

**Step 2: SSH Authentication Setup**
```bash
# Generate SSH key on local machine
ssh-keygen -t rsa -C "admin@test-cloud.com"

# Display public key content
cat ~/.ssh/id_rsa.pub

# Copy public key content
```

**Step 3: Register SSH Key in CSR**
1. Return to Cloud Source Repositories console
2. Access repository settings (three-dot menu)
3. Select "Manage SSH Keys"
4. Click "Register SSH Key"
5. Enter descriptive key name
6. Paste public key content
7. Click "Register"

**Step 4: Clone and Work with Repository**
```bash
# Clone the repository
git clone ssh://admin@test-cloud.com@source.developers.google.com:2022/p/test-project/r/my-new-repo

# Navigate into repository
cd my-new-repo

# Create sample file
touch newfile.txt

# Stage and commit changes
git add newfile.txt
git commit -m "Add initial file"
git push origin master
```

**Step 5: Verify Changes**
- Refresh Cloud Source Repositories console
- Confirm new file appears in repository view

### Demo Scenario: Mirroring External Repository

**Bitbucket Repository Mirroring:**
1. Create new CSR repository
2. Select "Connect an external repository"
3. Choose "Bitbucket" as source
4. Authenticate with Bitbucket credentials
5. Grant Google Cloud repository access permissions
6. Select specific repositories to mirror
7. Complete connection setup

**Verification Steps:**
1. Make changes in original Bitbucket repository
2. Confirm automatic sync to CSR
3. Test repository cloning with different authentication methods

### Expert Insight

**Real-world Application:** Cloud Source Repositories excel in GCP-centric development workflows where teams need secure, private Git hosting integrated with Google Cloud's CI/CD pipelines. Use mirroring capabilities to maintain single source of truth in GitHub while enabling Cloud Build access. Implement Pub/Sub notifications for automated security scanning or deployment triggers.

**Expert Path:** Master CSR by understanding Google Cloud's identity and access management integration. Learn advanced `gcloud` commands for automated repository management. Explore integration patterns with Cloud Build, Cloud Functions, and other Google Cloud services. Study authentication method selection based on use cases (SSH for developer machines, manual credentials for CI/CD systems).

**Common Pitfalls:** Avoid using mirrored repositories as primary development locations due to read-only limitations. Don't share manual credentials widely as they provide project-wide access. Remember that sensitive file warnings are advisory only - implement pre-commit hooks for strict enforcement. Be aware of the deprecation timeline and plan migration to newer Google Cloud repository solutions like Cloud Build repositories or Artifact Registry Git repositories.
