## Introduction to System Logging and Log Management

System logging and log management in Linux refers to the process of capturing, storing, and managing messages generated by the system, applications, and kernel. These logs are crucial for monitoring system health, troubleshooting issues, security analysis, and auditing. The primary system log services handle the collection, handling, filtering, and storage of logs in organized files.

Key concepts include understanding log rotations to manage file sizes, using commands to access logs securely, and configuring services like rsyslogd for centralized logging. Logs provide insights into system boot processes, user activities, application errors, and kernel events.

### Importance of Log Management
- **Troubleshooting**: Logs help diagnose problems like boot failures, authentication errors, or hardware issues by reviewing historical data.
- **Security Monitoring**: Track unauthorized access attempts, policy violations, and system integrity.
- **Performance Analysis**: Monitor resource usage and identify bottlenecks through application and kernel logs.
- **Compliance**: Maintain audit trails for regulatory requirements in environments like Red Hat-based systems (e.g., RHEL, CentOS).

⚠️ **Warning**: Logs may contain sensitive information like user credentials or system paths. Always ensure proper permissions and consider log encryption for production environments.

## System Log Daemons: journald and rsyslogd

The system log daemons are responsible for managing log generation, routing, and storage. In Linux, especially Red Hat-based distributions, two main services handle this:

### systemd journald (Journal Daemon)
- **Description**: Provides a binary logging system that stores log entries in a structured format. It is reliable for handling logs from system startup, kernel modules, and applications.
- **Features**:
  - Binary storage for faster retrieval.
  - Supports querying with filters (e.g., by time, priority).
  - Retains logs until explicitly cleared or disk space is managed.
- **Location**: Logs are stored in `/var/log/journal/` and managed by systemd.
- **Usage**: Use `journalctl` to view logs. It is the default in modern systemd-based systems.

### rsyslogd (Syslog Daemon)
- **Description**: An extended syslog daemon that supports advanced features like remote logging, message filtering, and network support. It replaces the older syslogd and is configurable for complex environments.
- **Features**:
  - **Reliable Network Support**: Uses TCP instead of UDP for improved message delivery.
  - **Message Filtering and Priorities**: Allows custom rules for routing logs based on facility and severity.
  - **Precise Timestamps**: Supports nanosecond-level timing for accurate event tracking.
  - **TLS Encryption**: Provides secure log transport for sensitive environments.
  - **SQL Database Support**: Can store logs in databases for centralized querying.
- **Configuration File**: `/etc/rsyslog.conf`
- **Location**: Outputs to text-based log files in `/var/log/`, depending on configuration.
- **Compatibility**: Supports both local and remote log handling, making it ideal for distributed systems.

| Feature               | systemd journald | rsyslogd          |
|-----------------------|------------------|-------------------|
| Storage Type         | Binary          | Text-based       |
| Remote Logging       | Limited        | Full support     |
| Encryption           | Basic           | TLS supported    |
| Filtering            | Moderate       | Advanced rules   |

✅ **Key Point**: Both daemons work together in modern systems; journald captures real-time logs, while rsyslogd handles routing and long-term storage.

## Log File Locations: /var/log/ Directory

The `/var/log/` directory is the primary location for system log files in Linux. It contains subdirectories and individual files where logs are stored based on their type and distribution.

### Directory Structure
- **General Logs**: Most logs reside directly in `/var/log/`.
- **Subdirectories**: Distros like Fedora/Red Hat may have subdirectories (e.g., for audit, samba, or package-manager logs).
- **Permissions**: Typically accessible only to root; use `sudo` or `su` for access.

Common log files and their purposes:
- System-generated messages from boot, kernel, applications.
- User activities, errors, warnings, and security events.
- Facility-specific logs (e.g., mail, cron jobs).

💡 **Pro Tip**: To list all log files: `ls -la /var/log/` or use `find /var/log/ -type f` to find files consuming space.

### Examples of Log Directories
- `/var/log/messages`: General system messages.
- `/var/log/boot.log`: Boot-time logs.
- `/var/log/secure`: Authentication and security-related events.
- Delegated subdirs like `/var/log/audit/` for SELinux audits.

## Key Log Files in Linux Systems

This section covers major log files found in `/var/log/`, including their locations, contents, and how to interpret them. Contents vary by distribution and installed services.

### General System Logs
- **messages** (`/var/log/messages`): Global log for system messages, including kernel, applications, and general errors. Contains startup processes, mail notifications, and user activities.
- **dmesg** (`/var/log/dmesg`): Kernel ring buffer logs detected during boot (e.g., hardware detection, module loading). Overwrites on reboot. Read with `dmesg` command.
- **boot.log** (`/var/log/boot.log`): Logs from system boot processes, including service startups and kernel messages at boot time.
- **syslog** (`/var/log/syslog`): Similar to messages but focuses on syslog-specific entries; contains kernel and application events.

### Authentication and Security Logs
- **secure** (`/var/log/secure`): Records authentication attempts, successful/failed logins, sudo usage, and security policy violations (e.g., PAM events). Key for intrusion detection.
- **wtmp** (`/var/log/wtmp`): Binary file tracking login sessions (who logged in, when). Read with `last` command for user activity history.
- **lastlog** (`/var/log/lastlog`): Records the last login time for each user. Binary format; use `lastlog` to query.
- **btmp** (`/var/log/btmp`): Failed login attempts. Binary; use `lastb` to view. Helps identify brute-force attacks.

### Application and Service-Specific Logs
- **cron** (`/var/log/cron`): Logs from cron jobs, including scheduled task executions and errors.
- **mail** (`/var/log/mail`): Email server logs (e.g., sendmail, postfix) for inbound/outbound emails and errors.
- **sudolog** (`/var/log/sudolog`): Sudo command executions and errors.
- **yum.log** (`/var/log/yum.log` in Red Hat): Package installation/removal logs (replaced by `dnf` in modern Fedora).
- **audit.log** (`/var/log/audit/audit.log`): SELinux and system auditing logs for policy violations and integrity checks.

### Kernel and Boot Logs
- **kern.log** (`/var/log/kern.log`): Kernel-specific messages, e.g., module loads, network interface events.
- **journal** (`/var/log/journal/`): Binary logs from systemd journald.

### Other Specialized Logs
- **ansible.log**: Ansible automation logs if installed.
- **httpd** or **apache2**: Web server logs (varies by distro/service).
- **samba**: Network share logs.

❌ **Common Pitfall**: These files grow over time. Monitor sizes with `du -sh /var/log/*` and implement log rotation to prevent disk exhaustion.

Messages vary by facility (e.g., auth, cron) and priority (e.g., error, warning). Read formats include timestamps, hostnames, processes, and error codes.

## Reading and Interpreting Logs

To access and analyze logs, use appropriate commands based on file type. Text files can use `cat`, `tail`, or `grep`, while binary files need specialized tools.

### Commands for Text-Based Logs
- `cat /var/log/messages`: View entire file (avoid for large files; use `head` or `tail` for snippets).
- `tail -f /var/log/messages`: Monitor real-time updates (useful for live debugging).
- `grep "error" /var/log/messages`: Filter specific patterns (e.g., errors).
- `dmesg | grep "usb"`: Kernel logs with filtering.
- `journalctl -r`: Recent systemd journal entries.
- `last`: Login history from `wtmp`.
- `lastlog`: Last login summary.

For binary files like `btmp` or `wtmp`:
- `lastb`: Failed login attempts.
- Provide permissions check first: `ls -la /var/log/wtmp`.

### Security Considerations
- Always use `sudo` for root-owned logs.
- Encrypt sensitive logs in production.
- Rotate logs to manage size and retention.

💡 **Expert Tip**: For quick troubleshooting, combine with `grep` for specific errors, e.g., `grep "denied" /var/log/secure` for access issues.

## Generating Test Logs Using Logger

To test log routing and configuration, use the `logger` command to send test messages.

### Basic Usage
- `logger "Test message"`: Sends a default INFO-level message to syslog.
- `logger -p kern.emerg "Emergency kernel test"`: Specifies facility (`kern`) and priority (`emerg`).

### Advanced Options
- `-f file`: Read message from file.
- `-i`: Include process ID.
- `-t tag`: Add a tag for identification.
- Generate varied messages: `logger -p auth.warn "Failed login test"` for security logs.

Messages appear in configured files based on rsyslog rules. Useful for verifying selector-action pairs.

### Demo Examples
- Send emergency: `logger -p local4.emerg "[EMERG] System alert"`
- Check results: `tail /var/log/messages` (if routed there).

✅ **Note**: Test in non-production environments to avoid false alarms.

## Rsyslogd Configuration: Selectors and Actions

Rsyslogd's configuration file (`/etc/rsyslog.conf`) defines how logs are handled using selectors (what to log) and actions (where/how to store).

### Selectors
Syntax: `facility.priority` (e.g., `mail.info`).
- **Facilities**: Log source types (e.g., `auth`, `cron`, `kern`, `mail`, `news`, `syslog`, `user`). Wildcard `*` for all.
- **Priorities**: Severity levels (ascending): `debug`, `info`, `notice`, `warning`, `err`, `crit`, `alert`, `emerg`. Use `=` for exact match, `!=` for exclusion.

### Actions
Define logging destination and behavior.
- **File Paths**: e.g., `/var/log/mail.log` for specific facilities.
- **Templates**: Add timestamps/tags with custom formats.
- **Filters**: e.g., `& Action` for complex rules.
- **Remote Logging**: `@host:port` or `@@host:port` (TCP).
- **Database**: Store in SQL for querying.
- **Processing**: Compress, encrypt, or rotate logs.

### Example Configuration Rules
- `authpriv.* /var/log/secure`: All authpriv facility logs to secure file.
- `mail.=info /var/log/mail.log`: Exact info-level mail logs.
- `*.* /var/log/messages`: All logs to messages (with sub-options like `!mail.none` to exclude).

| Selector Example   | Action Example               | Effect                                   |
|--------------------|------------------------------|------------------------------------------|
| mail.info         | /var/log/mail.log           | Mail info+ to mail.log                  |
| *.emerg           | /var/log/emerg.log          | All emergency to emerg.log              |
| kern.!info        | /var/log/kern_excl.log      | Kernel except info to file              |

⚠️ **Warning**: Incorrect rules can flood logs or miss critical events. Test changes in staging.

## Log Rotation Management

Log rotation prevents files from growing infinitely by compressing, moving, or deleting old logs. Managed by `logrotate` in Linux.

### How It Works
- Automatically rotates based on time/size thresholds.
- Compresses old logs (e.g., `.gz` files).
- Retains specified number of files (e.g., weekly for 4 weeks).

### Configuration
File: `/etc/logrotate.conf` or service-specific files in `/etc/logrotate.d/`.
- **Options**: 
  - `weekly`: Rotate weekly.
  - `rotate 4`: Keep 4 rotated files.
  - `compress`: Compress old logs.
  - `create`: Make new log file post-rotation.
  - `size 100M`: Rotate when >100MB.

### Example Config for `/var/log/messages`
```
/var/log/messages {
    weekly
    rotate 4
    compress
    create 0644 root root
}
```

### Checking Disk Usage
- `du -sh /var/log/*` to find large files.
- Use `logrotate -d /etc/logrotate.d/syslog` to test dry-run.

💡 **Expert Path**: For advanced setups, integrate logrotate with cron for custom schedules. Monitor space with alerts to prevent issues.
