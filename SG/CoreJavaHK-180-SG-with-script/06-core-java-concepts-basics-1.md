## Common Terminology Used in Programming Languages

### Overview
Programming languages share fundamental terminology that describes the process of developing, translating, and executing code. Understanding these core concepts is essential before diving into language-specific syntax. These terms explain how human-readable code gets transformed into machine-executable instructions through various stages and tools.

### Key Concepts/Deep Dive
- **Source Code**: A high-level code written by programmers using English-like characters and mathematical operators for human understanding. Examples include:
  - C language: `add.c`
  - C++: `add.cpp`
  - Java: `add.java`
  - C#: `add.cs`
  - Python: `add.py`
- **Compiled Code**: Low-level code generated by a compiler from source code, understandable to either the operating system (machine language) or intermediate software (bytecode).
  - C/C++: Object code (`.obj` files)
  - Java: Bytecode (`.class` files)
  - .NET: MSIL code (`.exe` assemblies)
  - Python: Bytecode (`.pyc` files)
- **Executable Code**: Code ready for direct execution, often stored as executable files (e.g., `.exe`, `.bin`) or run via interpreters without separate storage.

#### Process Flow
```diff
Source Code → Compilation → Compiled Code → Execution → Output
```

- **Compilation**: Process of translating source code into compiled code using a compiler. This creates platform-specific or intermediate code.
- **Execution**: Running compiled code to produce output. May involve interpreters for further translation to machine code.
- **Compiler**: Translator software that converts source code to compiled code and saves it as a file.
- **Interpreter**: Translator software that converts source or compiled code directly to machine code at runtime, without saving compiled forms.

#### Programming Language Classifications
- **Compiled Programming Language**: Uses only compiler (e.g., C, C++). Source code → machine code → executable file for repeated runs.
- **Interpreted Programming Language**: Uses interpreter directly on source code (e.g., PHP, JavaScript). No compilation step.
- **Both Compiled and Interpreted (Hybrid)**: Uses compiler to generate bytecode, then interpreter for execution (e.g., Java, Python, .NET).

```bash
# Example: Compiling a Java program
javac Add.java
# Generates Add.class (bytecode)
```

```bash
# Example: Running Java program
java Add
# Output: Addition result (e.g., 30)
```

> [!NOTE]
> In pure compiled languages like C, you need linkers to create executable files from object code.

## Platform Dependency vs Platform Independency

### Overview
Platform dependency means a program compiled on one operating system works only on that system. Platform independency allows the same compiled program to run on any operating system without recompilation.

### Key Concepts/Deep Dive
- **Platform Dependent**: C/C++ programs generate OS-specific machine code (executable files). Compile on Windows → Windows executable; recompile for Linux.
- **Platform Independent**: Java uses bytecode as an intermediate language. Bytecode runs on any OS with JVM installed.

| Aspect | Platform Dependent | Platform Independent |
|--------|-------------------|---------------------|
| Languages | C, C++ | Java |
| Compiled Output | Machine code (.exe, .obj) | Bytecode (.class) |
| Recompilation Needed | Yes, for each OS | No, runs everywhere with JVM |
| Execution Tool | OS directly | JVM |

## How Java Achieves Platform Independency

### Overview
Java achieves platform independency through bytecode, an intermediate compiled form that JVM translates to machine code at runtime.

### Key Concepts/Deep Dive
- **Java Compilation Process**:
  ```diff
  Java Source (.java) → Compiler (javac) → Bytecode (.class) → JVM → Machine Code → Execution
  ```
- JVM acts as an interpreter, converting bytecode to OS-specific machine code.
- No direct machine code generation; bytecode is portable across platforms.
- Example Workflow:
  ```bash
  # Compile once
  javac HelloWorld.java
  # Run anywhere with JVM
  java HelloWorld  # Works on Windows, Linux, macOS
  ```

## Types of Java Softwares

### Overview
Java provides different software bundles for various development and execution needs.

### Key Concepts/Deep Dive
- **JDK (Java Development Kit)**: For developers; includes tools for compilation, debugging, and development.
- **JRE (Java Runtime Environment)**: For running Java applications; minimal runtime without development tools.
- **JVM (Java Virtual Machine)**: Core runtime that interprets bytecode into machine code.

| Software | Purpose | Components |
|----------|---------|------------|
| JDK | Full development suite | Compiler, debugger, JVM, libraries |
| JRE | Application execution | JVM, runtime libraries |
| JVM | Bytecode interpretation | Platform-specific interpreter |

## JDK, JRE, JVM

### Overview
These three components form the Java ecosystem. JDK for building, JRE for running in production, JVM as the execution engine.

### Key Concepts/Deep Dive
- **JDK**: Contains JRE + development tools (e.g., `javac`). Version-specific differences in Java 11.
- **JRE**: Contains JVM + core libraries for application execution.
- **JVM**: Translates bytecode to machine code; varies by platform but ensures consistent Java behavior.
- Version Changes in Java 11:
  ```diff
  - Some internal libraries reorganized for modularity
  + Improved startup performance
  ```

```bash
# JDK contains JRE, which contains JVM
# Check version
java -version
javac -version
```

## JDK Folder Structure Differences

### Overview
JDK folder structure evolved with modularity in Java 9 and significant changes in Java 11, affecting tool locations and library organization.

### Key Concepts/Deep Dive
- **Java 8 and Earlier**: `bin` folder contains `javac`, `java`; `lib` has jars.
- **Java 9+**: Introduced modularity; `jmods` folder for modules.
- **Java 11 Changes**:
  ```diff
  - Removal of some tools (e.g., Java EE APIs became separate)
  + Enhanced security features
  - Simplified structure for cloud deployments
  ```
- Key Folders:
  - `bin`: Executables (`javac`, `java`)
  - `lib`: Core libraries
  - `include`: Native headers
  - `jre` (until Java 8)

| Version | Key Change | Impact |
|---------|-----------|---------|
| Java 9 | Modules introduced | Better dependency management |
| Java 11 | Tool consolidation | Reduced bundle size |

## Setting Java Home, Class Path, and Module Path

### Overview
Environment variables ensure system-wide access to JDK components, enabling compilation and execution from any directory.

### Key Concepts/Deep Dive
- **JAVA_HOME**: Points to JDK installation root (e.g., `C:\Program Files\Java\jdk-11.0.1`).
- **CLASSPATH**: Specifies additional library paths for class loading.
- **MODULE PATH**: For modular Java applications, points to module locations.
- Setting Example (Windows):
  ```cmd
  set JAVA_HOME=C:\Program Files\Java\jdk-11.0.1
  set PATH=%PATH%;%JAVA_HOME%\bin
  set CLASSPATH=.;%JAVA_HOME%\lib\tools.jar
  ```

## Difference Between Path and Class Path

### Overview
PATH defines executable locations, while CLASSPATH specifies Java class/library file locations.

### Key Concepts/Deep Dive
- **PATH**: Allows OS to find executables (e.g., `javac`, `java`).
- **CLASSPATH**: Tells JVM where to find class files and libraries at runtime.
- Key Differences:
  ```diff
  - PATH: OS-level, for executable programs
  + CLASSPATH: JVM-level, for .class/.jar files
  ```
  | Variable | Scope | Example Setting | Purpose |
  |----------|-------|----------------|---------|
  | PATH | OS | `%JAVA_HOME%\bin` | Locate `java.exe` |
  | CLASSPATH | JVM | `.lib\myjar.jar` | Load custom classes |

> [!IMPORTANT]
> Always include `.` in CLASSPATH to allow current directory class loading.

## Summary

### Key Takeaways
```diff
+ Source code is human-written, high-level code (e.g., .java files)
+ Compilation converts source to compiled code (e.g., bytecode in Java) using compilers
+ Interpretation translates bytecode to machine code at runtime for execution
+ Java achieves platform independency via JVM interpreting bytecode across OS
- C/C++ are platform-dependent; require recompilation per OS
! Always set JAVA_HOME, PATH, and CLASSPATH for JDK usage
! Java 11 introduced significant structural changes to JDK
```

### Expert Insight

#### Real-world Application
In enterprise Java development, understanding compilation vs. interpretation is crucial for optimizing deployment. For high-performance applications, consider JVM tuning or AOT (Ahead-of-Time) compilation tools like GraalVM to convert bytecode to native binaries, gaining C++-like speed while maintaining Java's portability.

#### Expert Path
Master containerized Java deployments (Docker) to leverage platform independency in cloud environments. Study JVM internals via tools like `jvisualvm` or `jmc`. Dive into bytecode engineering with ASM framework for advanced optimization. Track JVM evolution through OpenJDK contributions to understand future modularity enhancements.

#### Common Pitfalls
- Forgetting to set PATH/CLASSPATH leads to "command not found" errors during compilation/execution.
- Mixing JDK versions (e.g., compiling with JDK 11, running on JRE 8) causes runtime incompatibilities.
- Assuming bytecode is human-readable like source code—it remains in JVM opcodes.
- Overlooking module path in Java 9+ projects results in "module not found" exceptions.
