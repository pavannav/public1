## Overview
In this session, we delve into advanced AWS networking by configuring custom Virtual Private Clouds (VPCs), establishing secure connectivity through VPC peering, and deploying EC2 instances with robust security groups. Participants learn how to design multi-zone subnets, set up internet gateways for public access, customize route tables for controlled traffic flow, and securely access private subnets via bastion hosts using SSH forwarding agents. This builds on foundational networking principles, emphasizing production-ready architectures that avoid overlapping networks and ensure client-compliant configurations.

## Key Concepts / Deep Dive
This section follows the instructor's flow, covering VPC fundamentals, subnet strategies, security implementations, and connectivity hurdles. Each concept is explained in depth with technical context, bullet-pointed steps, and practical applications.

### VPC Creation and Customization
Virtual Private Clouds (VPCs) are isolated virtual networks in AWS, allowing you to define IP address ranges, subnets, and security. Unlike default VPCs, customized VPCs give you control over network architecture, which is essential for production environments where client requirements dictate non-overlapping static routes and multi-availability zone deployments.

- **Key Components**: CIDR blocks (e.g., 10.1.0.0/16), subnets (public/private), internet gateways (IGWs), and route tables.
- **Customization Tips**: Always delete default VPCs in production to enforce client-specified schemes. Use /24 for VPC CIDR if multiple subsidiary networks are needed, ensuring non-overlapping ranges to prevent peering failures.
- **Security Note**: Default VPCs are acceptable for testing but never for client projects, as they may lack required customization.

### Subnet Configuration in Multiple Availability Zones
Subnets segment VPCs into smaller networks, typically public (for bastion hosts with IGW access) and private (for application/DB servers). Distributing subnets across availability zones (AZs) ensures high availability.

- **CIDR Allocation**: Use /28 for bastion subnets to minimize IP waste; /26 for application subnets covering multiple VMs.
- **AZ Assignment**: Rotate AZs (e.g., us-east-1a, 1b, 1c) to avoid single-point failures. Non-overlapping CIDRs across AZs (e.g., 10.1.0.0/28 for 1a, 10.1.0.16/28 for 1b).
- **Best Practice**: All subnets start in the main route table; customize only for specific traffic needs.

> [!WARNING]
> Overlapping subnet ranges will prevent VPC peering. Always verify CIDRs against on-premise or peer VPCs.

### EC2 Instance Deployment with Security Groups
EC2 instances are virtual servers; security groups act as firewalls controlling inbound/outbound traffic.

- **Security Group Rules**: Stateful (response traffic auto-allowed). For bastion hosts: SSH (22) from anywhere (but recommend IP whitelisting). For private instances: SSH only from bastion SG.
- **Multi-NIC Considerations**: Attach multiple network interfaces if needed, but restrict SG by interface or IP.
- **Key Pair Management**: Use RSA keys for SSH; copy public key to instances for authentication.

```bash
# Example SSH key generation
ssh-keygen -t rsa -f networking-key

# Add key to SSH agent (for forwarding)
ssh-add networking-key
```

### Internet Gateway (IGW) Setup
IGWs connect VPCs to the internet, enabling public access for bastion hosts.

- **Attach to VPC**: Select the IGW and attach to your custom VPC (not default).
- **Route Table Update**: Add routes to IGW for 0.0.0.0/0 in the main route table.
- **Limit Exposure**: Only assign public IPs to necessary instances (e.g., bastions).

### VPC Peering Connection
Peering links two VPCs for private communication, bypassing public internet and DX circuits.

- **Requester/Accepter**: Same account uses "accept" workflow. Intra-region peering adds no latency.
- **Route Propagation**: Default denies cross-VPC traffic; manually add routes to peer VPC CIDRs via peering connection.

### Route Table Customization
Route tables define traffic paths within/defining VPCs.

- **Main vs. Custom**: Main applies to new subnets; create custom for specific rules (e.g., peering).
- **Propagation**: Enable for dynamic routes (e.g., VPN) or manual entries for static peering/DX.

> [!NOTE]
> Associate subnets explicitly to custom route tables; local routes always included.

### Bastion Host Access and SSH Forwarding Agent
Bastion hosts (jump boxes) provide secure entry to private subnets.

- **Setup**: Public subnet with IGW; tight SG permitting SSH only.
- **SSH Forwarding**: Load private key into SSH agent; forward to bastion, then to private instances. Avoid copying keys locally on bastion (risky).
- **Methods**: 
  1. PuTTY with PPK (for Windows).
  2. PowerShell/Linux with agent (recommended).
  3. Copy key to bastion (not best practice due to compromise risks).

## Lab Demos
Detailed steps with commands, configurations, and troubleshooting from the session.

### Demo 1: Creating VPCs and Subnets
1. **Create First VPC**: Name "Networking A", CIDR 10.1.0.0/24, tenancy default.
   ```yaml
   Name: Networking A
   CIDR: 10.1.0.0/24
   Tenancy: Default
   ```
2. **Create Second VPC**: Name "Networking B", CIDR 10.1.0.0/24 (same region, different cluster).
   ```yaml
   Name: Networking B
   CIDR: 10.1.0.0/24
   Tenancy: Default
   ```
3. **Subnet Creation for Networking A**:
   - Subnet 1: 10.1.0.0/28, AZ us-east-1a, public.
   - Subnet 2: 10.1.0.16/28, AZ us-east-1b, private.
   - Subnet 3: 10.1.0.32/28, AZ us-east-1c, private.

   | Subnet Name | CIDR | AZ | Type |
   |-------------|------|----|------|
   | Bastion | 10.1.0.0/28 | us-east-1a | Public |
   | Private A | 10.1.0.16/28 | us-east-1b | Private |
   | Private B | 10.1.0.32/28 | us-east-1c | Private |

4. **Subnet Creation for Networking B**: Similar, but for Network B with /26 CIDRs.

### Demo 2: Deploying EC2 Instances with Security Groups
1. **Create Key Pair**: RSA, name "networking-key".
2. **Bastion Host**:
   - VPC: Networking A
   - Subnet: 10.1.0.0/28 (public)
   - SG: SSH from anywhere (customize to IP for security).
   ```yaml
   Inbound Rules:
   - Type: SSH (22), Source: Anywhere (0.0.0.0/0)
   Outbound: All traffic allowed (default)
   ```
3. **Private Instances**:
   - Subnet: Private A/B
   - SG: SSH from Bassian SG only.
   ```yaml
   Inbound Rules:
   - Type: SSH (22), Source: SG of Bastion Host
   - Type: ICMP (ping), Source: SG of Bastion Host
   Outbound: All traffic allowed
   ```
4. **Launch**: Use free tier T2.micro; assign public IP for bastion; dynamic for others.

### Demo 3: VPC Peering and Route Table Updates
1. **Create Peering**: Requester VPC A, Acceptor VPC B; accept request.
2. **Custom Route Tables**:
   - Table for Private A: Add route to 10.1.0.0/24 via peering ID.
   - Table for Private B: Similar for connectivity.
3. **Associate Subnets**: Link private subnets to custom tables.
4. **Verify**: Ping across VPCs (ensure SG allows).

### Demo 4: SSH Access via Bastion Host
1. **Local Setup**: Add key to agent.
   ```bash
   cd Downloads
   ssh-add networking-key
   ```
2. **Connect to Bastion**: Use public IP/DNS.
   ```bash
   ssh -A ec2-user@<bastion-public-ip>
   ```
3. **Jump to Private**: From bastion.
   ```bash
   ssh ec2-user@<private-ip>
   ```
   - Troubleshoot: Ensure key added to bastion; SG correct; no permission issues (chmod not needed if using agent).

## Summary

### Key Takeaways
```diff
+ VPC Peering allows secure cross-VPC communication without public internet exposure.
+ Subnet overloading can break peering—use non-overlapping CIDRs (e.g., /28 for small, /26 for larger).
- Avoid copying private keys to bastions; use SSH agents for secure forwarding.
! Security groups control traffic at instance level—stateful and granular.
+ Route tables must be customized manually for peering; local routes are implicit.
- Default VPCs are unsuitable for client projects—always create customized ones.
```

### Quick Reference
- **VPC CIDR**: 10.1.0.0/24 for multi-service; ensure no overlaps.
- **SSH with Agent**: `ssh -A ec2-user@<ip>` to forward keys.
- **Peering Route**: Destination <peer-CIDR>, Target <peering-ID>.
- **IGW Attach**: `aws ec2 attach-internet-gateway --vpc-id <vpc> --internet-gateway-id <igw>`.

### Expert Insight

#### Real-world Application
In production, VPC peering connects development/staging/prod VPCs for shared databases or APIs. Transit Gateways extend this for multiple regions. Use VPN over peering for on-premise links, ensuring minimal latency for applications like video streaming or real-time analytics.

#### Expert Path
Master CIDR planning (RFC 1918 for private)—practice with 3+ VPC peering scenarios. Automate via AWS CDK/Terraform: deploy stacks with dynamic route tables. Certify AWS Advanced Networking Specialty for complex architectures like global hybrid clouds.

#### Common Pitfalls
- Overlapping CIDRs: Test peering early; fails silently.
- SG Misconfigurations: SSH from bastion only—audit logs for unauthorized access.
- Route Table Errors: Missing local routes cause isolation; double-check associations.
- Agent Key Expiration: Session-based; restart agent if connections fail.

#### Lesser-Known Facts
- VPC peering is transitive with Transit Gateway; direct peering remains non-transitive.
- IGWs manage SNAT for outbound traffic; internal traffic bypasses it.
- SSH agents cache keys in memory; no disk exposure, but sessions expire keys.

#### Advantages and Disadvantages
**Advantages**:
- Isolated networks reduce attack surfaces.
- Peering enables cost-effective private connectivity (vs. DX).
- Multi-AZ boosts reliability.

**Disadvantages**:
- No default transitive routing; requires manual config.
- Region-locked peering (for inter-region, use TGW).
- Security groups add overhead for complex rules.
