## RSA Key Authentication and Bastion Host Management

### Overview
This session explores the fundamental concepts of SSH key-based authentication using RSA cryptography, focusing on bastion host implementation and multi-instance key management in AWS environments. The discussion covers the encryption/decryption process, public-private key pair usage, and practical bastion host configurations for secure access to private subnet resources.

### Key Concepts / Deep Dive

#### RSA Key Encryption/Decryption Process
The RSA authentication process involves three distinct phases:
- **Key Generation**: Creates matched private-public key pairs using RSA algorithm
- **Authentication Flow**: Server encrypts challenge with client's public key, client decrypts with private key
- **Mutual Authentication**: Both client and server authenticate each other through encrypted communications

**Server Authentication of Client:**
```
Client Private Key → Encrypt with Server's Public Key → Send to Server
Server receives packet → Decrypts with own Private Key → Verifies Client's Public Key
Algorithm processes keys → Matches against server's database → Authentication succeeds/fails
```

**Client Authentication of Server:**
```
Server Private Key → Encrypt with Client's Public Key → Send to Client  
Client receives packet → Decrypts with own Private Key → Verifies Server's Public Key
Algorithm processes keys → Matches against client's database → Authentication succeeds/fails
```

#### Bastion Host Implementation

**Production Environment Considerations:**
- Limited bastion host connections maintain security boundary
- Single key pair used across multiple instances
- Platform-agnostic access through Putty, PowerShell, or Linux CLI

**Key Management Strategies:**
```bash
# Create blank key file
touch my-private-key.pem

# Open editor and paste private key content
nano my-private-key.pem
# OR
vim my-private-key.pem

# Set proper permissions (owner read-only)
chmod 400 my-private-key.pem

# SSH connection using key file
ssh -i my-private-key.pem ec2-user@<private-ip>
```

**Windows Platform Access Options:**
- **PuTTY**: Convert .pem to .ppk format for SSH key usage
- **PowerShell**: SSH client with agent forwarding capabilities
- **WSL/Linux Subsystem**: Native Linux SSH client functionality

**SSH Agent Forwarding Workflow:**
```bash
# Add key to SSH agent
ssh-add my-private-key.pem

# Forward key to bastion host during login
ssh -A ec2-user@<bastion-public-ip>

# Access multiple private instances without repeated key authentication
ssh ec2-user@<private-ip-1>
ssh ec2-user@<private-ip-2>
```

#### Multi-Instance Key Management

**AWS Key Pair Usage:**
- Single key pair enables access to unlimited EC2 instances
- Public key stored by AWS across infrastructure
- Encrypted private key requires proper access controls

**Permission Structure for Multi-Instance Access:**
```bash
# Owner gets read-only access (4)
# Group gets no access (0)  
# Others get no access (0)
chmod 400 private-key.pem  # 400 = r--------

# OR for owner and group read-only
chmod 440 private-key.pem  # 440 = r--r-----
```

## VPC Peering Configuration

### Overview
VPC peering creates private network connections between AWS virtual networks, enabling secure cross-VPC communication without internet gateway dependencies. This section covers the complete peering implementation process, route table modifications, and connectivity validation between peered virtual networks.

### Key Concepts / Deep Dive

#### VPC Peering Fundamentals
**Core Characteristics:**
- Private IP-based communication between VPCs
- No transitive routing (peer-to-peer only)
- Same AWS account peering simplified
- Cross-region peering supported
- No bandwidth charges within same region

**Peering Status Types:**
- **Pending Acceptance**: Initial state after request creation
- **Active**: Successfully established connection
- **Rejected**: Acceptor denied the request
- **Expired**: Request timed out without acceptance

#### Peering Implementation Process

**Requestor Actions:**
```bash
# Navigate to VPC Dashboard
- Select "Peering connections"
- Click "Create peering connection"
- Provide descriptive peering name
- Select requestor VPC
- Select acceptor VPC (same account)
- Create connection
```

**Acceptor Actions:**
```bash
# Navigate to Acceptor VPC Dashboard  
- Select "Peering connections"
- Find connection by name/ID
- Click "Actions" → "Accept"
- Confirm acceptance
```

#### Route Table Modifications

**Network Topology Example:**
- VPC-A: Networking-A (10.1.0.0/16)
  - Private Subnet A: 10.1.1.0/28 (us-east-1a)
- VPC-B: Networking-B (10.1.2.0/16) 
  - Private Subnet B: 10.1.2.64/26 (us-east-1b)

**Route Table Configuration for VPC-A:**
```bash
# Local route (automatic)
10.1.0.0/16 → local

# Cross-VPC route via peering connection
10.1.2.0/16 → pcx-[peering-connection-id]
```

**Route Table Configuration for VPC-B:**
```bash
# Local route (automatic)
10.1.2.0/16 → local

# Cross-VPC route via peering connection

```

#### DNS Resolution Options
- **Disabled by Default**: Private DNS names not resolvable across peering
- **Enable if Required**: Allow DNS hostnames resolution (additional latency consideration)
- **Local Resolution Preferred**: Minimize cross-region DNS lookups

## Route Table Customization

### Overview
Route table customization ensures proper traffic routing in VPC peering scenarios, defining specific paths for cross-VPC communication while maintaining security boundaries and preventing unintended network exposure.

### Key Concepts / Deep Dive

#### Route Table Association Strategy

**Main Route Table Limitations:**
- All subnets associate automatically
- Promiscuous routing behavior
- Security boundaries not enforced
- Limited granular control

**Custom Route Table Benefits:**
- Subnet-specific associations
- Granular routing control
- Enhanced security isolation
- Compliance with least privilege principles

**Association Process:**
```bash
# Create custom route table
- Navigate to "Route Tables" subsection
- Click "Create route table"
- Select VPC association
- Provide descriptive name

# Associate specific subnets
- Select route table
- Choose "Subnet associations"
- Associate specific private subnets
- Remove from main route table if needed
```

#### VPC Transit Patterns

**Hub-and-Spoke Architecture:**
```
Spoke VPC-A → Hub VPC → Spoke VPC-B
```

**Peering Implementation:**
```
VPC-A Router ↔ VPC-B Router (Direct peer)
```

**Route Propagation Considerations:**
- Static routes manually configured
- No dynamic BGP routing in peering
- Explicit route destination definition required
- No transitive routing capability

### Expert Insight

**Real-world Application:** 
In enterprise environments, VPC peering enables segregated application architectures where web servers in one VPC communicate privately with database servers in another VPC. Bastion hosts in public subnets provide controlled administrative access while maintaining security isolation for sensitive workloads.

**Expert Path:** 
Master bastion host agent forwarding and automated key management scripts. Understand Network ACL vs Security Group precedence in cross-VPC scenarios. Implement automated VPC peering testing through Infrastructure as Code pipelines.

**Common Pitfalls:** 
Unnecessarily enabling DNS resolution across peering causes performance degradation. Overly permissive security groups create security gaps. Failing to update route tables during architectural changes leads to connectivity blackouts.

**Lesser-Known Facts:** 
SSH agent forwarding can create security loopholes if compromised bastion hosts are accessed by unauthorized users. Route table metric precedence affects traffic flow in overlapping CIDR scenarios.

**Advantages of VPC Peering:** 
- Lowest latency for cross-VPC communication
- Higher security than internet-based connections
- No bandwidth costs within same AWS region
- Simplified network architecture without transit gateway complexity

**Disadvantages of VPC Peering:** 
- No transitive routing limits scalability for complex architectures
- Manual route table management burden for large environments
- Cross-region peering incurs bandwidth costs
- Maximum of 125 peering connections per VPC creates architectural constraints
