Building on our previous sessions of Runners and Executors, we will now focus on how these
components work together and orchestrate the entire process. So far we discussed about the
Runner and Executor, but what is this GitLab Server? GitLab Server is an open-source DevOps
platform that allows you to manage your entire software development lifecycle from a single
location. It provides features like version control, issue tracking, CI/CD pipelines,
package registries, and much more. So it comes in two flavors, GitLab SaaS and the GitLab
self-managed. Both GitLab SaaS and the self-managed offer the similar capabilities, but they cater to
different needs and come with their own set of advantages and disadvantages. So talking about
the advantages, so within GitLab SaaS, it is easy to use because there is no need of any
infrastructure setup or maintenance. One can get started quickly with minimal technical expertise
and it is also scalable with pay-as-you-go pricing scales seamlessly with your needs.
And you don't need to worry about managing the server capacity. GitLab also manages the
infrastructure, ensuring high uptime and redundancy. And it also gives you automatic
updates, guaranteeing access to the latest features and security patches. And finally,
GitLab handles the security patching as mentioned earlier and also the vulnerability management.
And talking about the pros within the GitLab self-managed instance, you get the full control
on customizing the platform to your specific needs and security requirements. There is no
vendor lock-in. You can easily migrate to another platform or host on your own infrastructure. You
get data privacy where you can maintain complete control over your code and data. And finally,
it is cost-effective for larger teams because potentially it is cheaper when compared with
SaaS for high volume usage. And talking about the challenges, so within GitLab SaaS,
it has limited customizations because you have less control over the platform configurations
compared to the self-hosted GitLab server. And even migrating to another platform can be a complex
and a costly affair. And potential data privacy concerns might arise because your code and data
are hosted on the GitLab infrastructure. And finally, the additional cost for storage and
resources might be expensive for larger teams or projects with high resource demands.
Within GitLab self-managed, you may require technical expertise to manage the infrastructure
and keep the platform updated. Scaling beyond your available infrastructure can be expensive
and complex. And finally, you are responsible for applying the security patches and also managing
any vulnerabilities. So now let's understand how all these three components work together like the
GitLab server, the GitLab runner, and the GitLab executor. And let's see how they interact during
the CI/CD process. So the first is the registration phase. The registration process is the first step
in setting up a GitLab runner. So during the registration, the runner is going to make a
POST request to the runner endpoint of the GitLab server. This request also includes a registration
token, which is a unique identifier provided by the GitLab server. The server is going to in turn
respond with a runner token upon successful registration. This runner token is then used
by the runner for future communications with the GitLab server. The second step is the GitLab
runner is going to request for the jobs and going to handle the jobs. So how is this going to work?
So once the runner is registered, it enters a loop where it continuously requests jobs from the
GitLab server. The runner is going to make a POST request to the request endpoint using the runner
token received during the registration phase. The server is going to respond with a job payload,
which also includes a job token. This job token is used by the runner to authenticate itself when it
performs actions related to the job. The third phase is the job execution phase.
So the job payload received from the server is passed on to the executor. So the executor,
as we discussed, is responsible for executing the job. It's going to use the job token to
authenticate itself when it needs to clone the resources or the sources and the artifacts from
the GitLab server. So the executor runs the job in an environment that is isolated from the runner
and the other jobs. And then we have the final phase where the executor is going to return the
job outputs and the status. So after the job is executed, the executor returns the job output
and the status. So the job token is still used to authenticate these actions. The runner then
updates the job output and status on the GitLab server using the same job token. So this is going
to allow the server to track the progress and the result of the job. So these are the key components
and how they interact within the GitLab CI/CD architecture. Thank you.
