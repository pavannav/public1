Let's talk about GitLab Auto DevOps.
Auto DevOps is a feature within GitLab that takes CI/CD automation to the next level.
It essentially configures and runs a complete CI/CD pipeline for you,
eliminating the need for manual setup.
So, what can we expect from the Auto DevOps?
So, Auto DevOps is going to analyze your project's programming language and framework
to determine the appropriate tools and settings.
It sets up stages for building, testing,
and deploying your application using industry best practices.
It's also going to integrate vulnerability scanning,
code quality checks, container scanning,
dependency scanning, and other security measures into the Pipeline.
It can also create review apps for testing
the application before merging to the source branch.
And post-merging, we can also deploy the application
either using continuous delivery or continuous deployment.
So, overall, Auto DevOps is a very powerful tool,
but it's not a one-size-fits-all solution.
So, depending on your project complexity and preferences,
you may need to further customize the pipeline.
So, let's see how this works at the project level.
Once the Auto DevOps is enabled at the CI/CD settings of the project,
it starts by detecting your code language
and uses CI/CD templates to build and run default pipelines.
It starts with the build job.
Auto DevOps leverages a pre-existing Dockerfile, if present,
or applies Heroku buildpacks to construct a Docker image
by identifying the coding language.
Regardless of the path taken,
a Docker image will be generated by the end of the build job.
Next is the testing stage.
During the test stage of the GitLab Auto DevOps,
your code gets put through its paces
using both your project's existing tests
and the power of Heroku buildpacks.
These handy tools simplify application testing
for various languages,
taking some of the heavy lifting off your shoulders.
However, it is important to note that
not all programming languages are currently supported in this stage.
Few of the supported languages are Ruby, NodeJS,
Maven, Gradle, Python, Java, Go, and few others.
So, this stage also includes code quality analysis,
static application security testing,
auto secret detection, container scanning, and many more.
So, metrics and reports of all these jobs
will be available for you to analyze
at the end of the pipeline.
Auto DevOps can also be configured
to deploy applications to Kubernetes clusters.
So, we can configure our GitLab project
with Kubernetes clusters.
So, it supports different Kubernetes clusters.
Few of them are Amazon Elastic Container Service,
Amazon Elastic Kubernetes Service,
Amazon EC2, Google Kubernetes Engine, and Bare Metal.
After the testing stage,
if deployment is configured,
the next stage is review apps.
The review stage is available during merge request.
So here, GitLab DevOps automatically deploys the changes
to a different environment
to review how the application works
before deploying to production.
It uses a Helm auto deploy chart
to deploy review applications.
This Helm chart can be modified as per the requirements.
After you deployed your code changes into a review app,
GitLab Auto DevOps takes things a step further.
It's not only just about seeing your application in action,
it's about ensuring it's secure
and ready for the real world.
That's where dynamic application security testing comes in.
So, think of DAST as a tester for your review apps.
It uses the open source OWASP ZAP Proxy tool
to scan your running applications,
searching high and low for any potential
security vulnerabilities or suspicious behavior.
And no need to worry about the manual setup,
Auto DevOps integrates DAST into the review apps workflow.
Once your app is deployed,
the ZAP Proxy gets to work mimicking real world attacks
and probing for any weaknesses.
If the ZAProxy uncovers any security vulnerabilities,
you'll get a detailed report
and this report pinpoints the vulnerabilities,
explains the severity,
and even offers suggestions for fixing them.
Depending on your Auto DevOps deployment configuration,
after merging the feature branch to the source branch,
Auto DevOps automatically deploys that
to the staging or production environment.
And when your application reaches production or staging,
Auto DevOps does not stop.
It runs the browser performance testing
to see how quickly your page loads.
This test compares your newer version to the older versions,
making sure it doesn't get sluggish over time.
But what about deployments?
Auto DevOps offers three different deployment options
for tailoring your application updates.
These are continuous deployment to production,
automatic staging, but manual production deployment,
and we also have something called
continuous deployment to production
using timed incremental rollouts.
So the first option,
which is the continuous deployment to production,
here, the updates are directly pushed
to the production environment whenever a code change.
So there is no manual intervention in the first option.
Well, talking about the second option,
updates are automatically flown to the staging environment.
It's going to wait for a manual approval
before hitting the production.
Optionally, GitLab features allow staged rollouts
in a controlled, incremental,
percentage-based fashion as well.
For example, if you have 10 pods and a 10% rollout,
one pod updates with the newer version
while others remain unchanged.
So manual release of each rollout stage
is triggered by clicking a play button.
The third option is the timed rollout option,
similar to the manual staging,
but each rollout stage automatically deploys
after a set delay, ideally in minutes.
So it is, again, customizable.
So you can customize the delay times
for added control over update progression.
And for example, this, the default setting,
the continuous deployment with five-minute intervals
between each rollout stage.

I don't think, you need to choose the approach
that best suits your needs
and you can ensure seamless and controlled updates
for your applications.
