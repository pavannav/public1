In the last session, we talked about GitLab CI/CD runners, which are essentially workers which
execute your pipeline jobs. So what are executors? Why do we need them? So runners themselves don't
directly run the jobs. They utilize something called an executor to actually carry out
the execution based on the specific needs of your project. And under the hood of each runner
lies a hidden engine called an executor. So what are these GitLab runner executors?
So you can think of an executor as a container within your runner. You can define the operating
system, tools, and the resources a job has to access. And they ensure that the jobs run in a
consistent and isolated environment. It's going to prevent conflicts and any unexpected behavior.
So GitLab offers a wide variety of built-in executors, each with its own pros and cons.
So choosing the right executor is crucial for ensuring efficient, secure, and reliable pipeline
execution. There are around 10 different types of GitLab Runner executors. So let's have a look at
some of the most widely used executors. So let's start with the Shell Executor. So this is a very
basic executor, which runs jobs directly on the runner's operating system using its shell like
Bash or PowerShell. It is very simple and lightweight. It's also suitable for quick
scripts or for testing basic functionality. And it has some disadvantages like it's going to share
the runner's operating system and resources. So it's going to always have a risk of conflicting
and unexpected behavior. And the results may also vary depending on the runner's environment
and the installed tools. And finally, sharing the resources might pose security risks if the
job requires different levels of access. The next executor we'll discuss is the
Docker Executor. So for greater isolation and reproducibility, Docker executors run jobs
inside a Docker container. These containers encapsulate the specific environments like the
operating system, libraries, tools needed for the job, guaranteeing consistency across multiple
machines. We can also choose from the countless pre-built Docker images with specific tools and
environments which are readily available. And of course, disadvantages are it requires
additional configuration and understanding of Docker concepts. You also need to run multiple
containers and it can consume more system resources on the runner. And finally, the
Docker containers often have restricted access to the runner's native resources.
And the third type of executor is a Kubernetes executor. If your project utilizes a Kubernetes
cluster, the Kubernetes executor lets you seamlessly run jobs as parts within those
clusters. This is going to leverage the cluster's resource management and scaling capabilities
for efficient execution and parallel processing. And of course, talking about the disadvantages,
it's going to have a very complex setup and requires complex maintenance because you need
to have familiarity with Kubernetes configuration and management in order to manage the executor
within a pod. And then it may not be feasible if you don't have a Kubernetes cluster. So,
there is a potential dependency on existing infrastructure. And finally, it's going to
have an increased learning curve because you need to first understand the best practices
and security considerations for running the jobs within Kubernetes.
The fourth type of executor is a virtual machine. So, if you need a completely clean slate for each
job, the VM executor is going to create a new virtual machine for each job execution,
guaranteeing isolation and eliminating potential contamination from previous ones. And of course,
it has its own disadvantages. One is it's going to have a very high resource usage because VMs
require significant disk space, memory, and CPU resources, potentially increasing your
infrastructure cost. And it's also going to have slower startup times because creating
and initializing VMs takes longer than starting containers, slowing down pipeline executions.
And finally, it also requires expertise in VM management and configuration.
The fifth type of executor which we will discuss is the SSH executor. This executor is going to
allow running jobs remotely on another machine via SSH connections. It is going to be useful
for utilizing specific resources available on a different machine or integrating with pre-existing
infrastructure. Talking about the challenges or disadvantages, it always has security concerns
because it requires secure SSH access and careful configurations to maintain jobs and system
security. It also relies on reliable network connectivity between the runner and the remote
machine. And finally, it has limited control and visibility because managing and monitoring jobs
running on a remote machine might be more challenging. And finally, the last type of
executor which we will discuss today is the custom executor. So GitLab offers additional
specialized executors like Parallel, Docker Machine, and even some custom ones which you can
develop yourself. So each is going to cater to a specific need and environments, ensuring flexibility
and catering to the diverse workflows. And talking about the disadvantages, it's also going
to have overhead and complexity because you have to create your own executor and start using it.
So you need to know how to develop it. And it's going to also increase the resource usage.
And the third one is it's going to have limited access to the underlying systems.
So these executors support different platforms and methodologies for building a project.
So this is an official table which shows the key facts for each executor, which will help you to
decide which one to use. And I would also recommend you to check out the official documentation
for more information. Thank you.
