In this session, we will explore the core concepts of a GitLab CI/CD pipeline.
It consists of four key components: pipeline, stages, jobs, and script.
So let's take an example to understand them.
A pipeline is an automated process capable of executing one or more jobs.
It's going to act as the blueprint for continuous integration and deployment.
It's going to ensure that the code changes flow smoothly through testing,
building, and the deployment phases.
The pipeline is defined using a YAML file,
and it is located within your repository right beside your codebase.
To uniquely identify different pipelines,
each pipeline can have an optional workflow named keyword.
So this name will be visible in the Pipelines tab of the GitLab repository.
Pipelines are triggered by events such as code pushes, Merge Requests, or manual actions.
In this example, the rules keyword within the workflow is going to trigger the pipeline
when there is a commit to the main branch.
We will talk more about triggers and rules in a later session.
The next core component is a job.
Jobs are the building blocks of the pipelines,
and you can have one or more jobs within the same pipeline.
Each job is associated with a runner.
It could be a GitLab-hosted or a self-hosted runner.
These runners are basically virtual machines,
which are responsible for running the job's instructions.
We can optionally define a tags keyword to specify the runner.
And in this example, the job will run on a Linux machine.
And if there is no tag keyword within your job,
the jobs will run on a default Runner.
So we will talk more about tags, runners, and the runner configurations in a later session.
So the core component of a job is called a script.
So the script contains the commands which need to be executed.
It is like the heart of the pipeline, defining the specific tasks which have to be performed.
So scripts can be written as individual commands,
as a shell script, or other supported languages.
It's going to offer flexibility in different automation tools.
So all the steps or commands within a script are executed sequentially.
And typically, scripts include commands which are used for building code,
running tests, deploying artifacts, performing security scans,
and also in some cases, generating documentation.
And each job can also have a before and or after scripts.
So these are optional scripts that run before or after the main script of a job,
providing opportunities for setup and teardown activities.
So the common use cases of the before script is to install the dependencies,
preparing test environment, or setting up some database connections.
And some common use cases of the after script include
cleaning up the resources after a pipeline execution is completed.
And if a pipeline has multiple jobs, by default,
all the jobs are going to be executed in parallel at the same time.
So to logically group jobs within a pipeline, we use a concept of stages.
So stages provide a logical grouping of jobs within a pipeline.
They represent distinct phases of the CI/CD process,
often aligned with the development milestones.
So stages execute jobs in a predetermined order, thereby enabling conditional execution,
allowing flexibility based on outcomes of the previous stages,
optimizing resource usage, and also preventing unnecessary actions.
So the common stages include building such as compiling the code,
you know, testing, running automated tests,
deploying the code from one environment to another environment.
And finally, you know, scanning for vulnerabilities using the security stage.
In this example, we have defined two stages, test and deploy.
And hence, the deploy job will start execution
only after the unit testing job successfully completes its execution.
And talking about pipelines.
In GitLab CI/CD, they are, I mean, it offers a wide range of pipelines.
And it has, you know, different configurations, which can be,
you know, used to tailor your software delivery process.
So we have the basic pipeline, which executes the jobs within a stage concurrently,
moving to the next stage only after all the jobs in the current stage are completed.
We have DAG pipelines, which can optimize efficiency by running jobs based on dependencies,
potentially reducing execution time compared to basic pipelines.
We have Merge Request pipelines, which trigger specifically for Merge Request,
ensuring Code Quality and consistency before merging branches.
We have something similar called as merge request pipeline,
which simulates the merge state of a merge request,
providing early insights into potential conflicts or issues.
Then we have merge train pipelines, which enact the Merge Request for sequential execution,
ensuring a controlled and orderly integration process.
We have parent-child pipelines, which are used for managing the complexity
by breaking down extensive pipelines into a parent pipeline that orchestrates multiple
child pipelines often used in a monorepo environment.
And finally, we have the multi-project pipelines,
which can coordinate pipelines across different projects,
fostering collaboration and streamlining cross-project dependencies.
So we can choose the pipeline, you know, that best suits your workflow,
team structure and project needs to achieve efficient and reliable software delivery.
