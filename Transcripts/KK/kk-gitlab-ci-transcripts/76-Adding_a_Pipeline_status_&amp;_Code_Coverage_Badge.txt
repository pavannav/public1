Let's talk about self-managed runners.
In our earlier sessions, we discussed runners, which are essentially virtual machines responsible for performing various tasks within a GitLab CI/CD pipeline.
For example, a runner can handle tasks such as cloning your repository, installing necessary software, and executing specific commands.
We have also discussed about the SAST runners, which are hosted and managed by GitLab.
These runners are fully integrated with GitLab.com and are enabled by default for all projects with no configuration required.
Each GitLab-hosted runner is essentially a fresh virtual machine that GitLab hosts for you.
So you have the options to choose from different operating systems like Linux, Windows, MacOS, and GPU-enabled SAST runners.
So what are self-managed runners?
Self-managed runners are machines that you deploy and manage yourself.
This gives you more control over the hardware, operating system, and the software tools that your workflow runs on.
Self-managed runners can be run on any operating system, including custom OS as well.
So here are some reasons why you might choose to use the self-hosted runners or the self-managed runners.
So the first one is about customizable environment.
You can set up your Runner with the specific software and resources needed for your projects, giving you complete control over the execution environment.
This can be useful for running jobs that require specific tools or libraries or that cannot be run on shared runners due to security restrictions.
In some organizations, strict security and compliance policies may prevent the use of GitLab-hosted runners for certain workflows.
So self-managed runners can be set up in a controlled environment, preventing conflicts and potential security vulnerabilities that could arise from sharing resources with other projects.
Depending on the demand for GitLab-hosted runners in your organization, there might be wait times for available runners.
So self-managed runners eliminate this wait time as they are dedicated to your projects.
You can scale your self-managed runner pool based on your project needs.
This can be particularly useful if you have many concurrent workflows or if you need to run multiple workflows in parallel.
Finally, self-managed runners can be placed in specific geographic locations to reduce latency for your CI/CD workflows, especially if you have a global user base or need to comply with data residency regulations.
In summary, while GitLab-hosted runners are convenient and suitable for many use cases, self-managed runners provide more control, flexibility, and customization options.
So the next step is to understand how to configure the runners in the context of GitLab CI/CD.
GitLab runners or the GitLab self-managed runners can be of three types.
So you can add them as shared runners which are available to all groups and projects within a GitLab instance.
This could be group runners which are available to all projects and subgroups in a group.
And finally, at the project level, so you can also add project runners which are associated with specific projects.
Typically, project runners are used by one project at a time, but we can also overwrite this configuration as well.
To set up a self-managed runner as a project runner, you can navigate to the desired project or repository's CI/CD settings.
There you will find an option to disable the shared runner for the project and also to create new project-level runners under the Runners section.
During the setup, you will have the opportunity to select the operating system of the machine where you will install the runner, add tags, and configure additional settings.
You can use tags to control the jobs a runner can run.
You can also check the Run Untagged Jobs box if you don't want to restrict this runner to specific jobs based on tags.
You can click Create Runner to generate a registration token to register the runner with GitLab.
Along with the token, the GitLab UI also provides steps to install the GitLab Runner application for the selected environments and architectures.
When setting up a runner for the first time on an operating system, it is essential to install the GitLab Runner application prior to registering the runner.
So, what are the steps required?
We first download the binary for your system using the curl command.
We give it permissions to execute using chmod.
We create a GitLab Runner user and also select a home directory.
And finally, we install and run the Runner as a service.
Now that the GitLab Runner application is installed, the next step is to register the runner.
During the registration process, you will be prompted to provide some details.
So, after you enter the GitLab Runner register command, the first thing it's going to ask for is your GitLab instance URL.
So, this ensures that the runner connects to the correct GitLab instance.
And then it's going to ask you for a name.
So, this is the name of the runner which is going to help you easily identify it within your CI/CD infrastructure.
And then you can also select the type of the executor which you want to use for this runner.
This determines how jobs will be executed such as using a Shell Executor or a Docker Executor.
And upon successful registration, all these configuration settings are automatically saved into a file named as config.toml.
This file serves as a central reference for managing the runner's behavior and settings.
To further customize your runner, we can edit this config.toml file located within the GitLab Runner installation directory on the machine.
So, this file allows you to define the runner name, the tags for identification, executors like Docker or Shell Executor to determine how the job runs.
You can also define resources like CPU, memory, and disk limits.
You can add environment variables for job executions.
And you can also add caching paths and many other configurations.
Finally, if you want to check the status of the setup, you can again go back to your GitLab project level CI/CD settings and check the runner status.
And you can also use the tags at a job level to select a specific runner from the list of all runners that are available for the project.
