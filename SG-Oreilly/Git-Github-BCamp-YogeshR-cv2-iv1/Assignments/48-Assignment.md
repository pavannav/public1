### Exercise 1.1: Understanding Pull Request Fundamentals
**Objective**: Grasp the core concept of pull requests as the gateway for open source contributions.

**Tasks**:
1. Analyze how PRs propose changes from forks/branches to main projects
2. Understand the maintainer review workflow: review work, provide feedback, decide on merging
3. Document the scenario: proposing committed changes from your fork to the original repository
4. Recognize the "one commit ahead" indicator as the trigger for creating pull requests

**Deliverable**: Pull request concept understanding document with contribution workflow overview.

---

### Exercise 1.2: Pull Request Initiation Process
**Objective**: Navigate the GitHub interface to initiate pull request creation.

**Tasks**:
1. Identify the "Contribute" button or "Pull requests" tab on your forked repository
2. Recognize the GitHub message indicating "branch is one commit ahead of main project"
3. Choose between clicking "Open pull request" directly or navigating through PR tab
4. Document the two initiation paths available for starting pull request creation

**Deliverable**: Pull request initiation process guide with interface navigation steps.

---

### Exercise 1.3: Repository Comparison Analysis
**Objective**: Understand the base/head repository comparison interface during PR creation.

**Tasks**:
1. Analyze the PR comparison view showing base repository (main project) vs head repository (your fork)
2. Verify the commit count and file change summary displayed by GitHub
3. Preview the exact differences in changed files using GitHub's diff viewer
4. Document how GitHub automatically identifies what changes will be proposed

**Deliverable**: Repository comparison analysis with diff preview documentation.

---

### Exercise 2.1: Pull Request Creation with Documentation
**Objective**: Create a well-structured pull request with clear title and comprehensive description.

**Tasks**:
1. Click "Create pull request" after reviewing the comparison
2. Add descriptive title: "Readme file typo fixed"
3. Write detailed description explaining the purpose and specific changes
4. Include concrete details about what was fixed (typos removed, CSS/JavaScript added)
5. Use the dropdown to select "Standard pull request" vs "Draft" status

**Commands** (PR creation content):
```
Title: Readme file typo fixed
Description: Fixed all typos and added CSS and JavaScript under technologies used section
Status: Standard pull request (not draft)
```

**Deliverable**: Successfully created pull request with comprehensive documentation and standard status.

---

### Exercise 2.2: Conflict Detection and Validation
**Objective**: Understand GitHub's automatic conflict detection and validation process.

**Tasks**:
1. Observe GitHub's automatic check for conflicts with the base branch
2. Verify the "no conflict" status that enables safe merging
3. Understand how GitHub prevents problematic merges before they occur
4. Document the importance of conflict-free PRs for successful contributions

**Deliverable**: Conflict detection validation documentation with merge safety verification.

---

### Exercise 2.3: Maintainer Review and Decision Process
**Objective**: Experience the maintainer perspective for reviewing and managing pull requests.

**Tasks**:
1. Switch to the main repository acting as the project maintainer
2. Navigate to the Pull requests tab to see incoming PRs
3. Review the PR description and examine the proposed changes
4. Document the maintainer's decision-making process: review, discuss, merge, or close

**Deliverable**: Maintainer review process documentation with decision workflow analysis.

---

### Exercise 3.1: Pull Request Merging Execution
**Objective**: Execute the merge process as a maintainer and understand merge documentation.

**Tasks**:
1. Click "Merge pull request" when satisfied with the proposed changes
2. Review the default commit message and description provided by GitHub
3. Confirm the merge action and observe the merge completion status
4. Document the merge metadata: who merged it, merge ID, and successful completion

**Commands** (merge process):
```
Action: Merge pull request
Commit message: Default (or custom)
Description: Default (or custom)
Result: Merged with user attribution and merge ID
```

**Deliverable**: Successfully merged pull request with documented merge metadata and completion status.

---

### Exercise 3.2: Related Issue Management
**Objective**: Connect pull request merges with related issue resolution workflows.

**Tasks**:
1. Navigate back to the Issues tab after successful PR merge
2. Identify related issues that are now resolved by the merged changes
3. Close the related issue manually using the "Close" button
4. Document the relationship between PR merges and issue resolution

**Deliverable**: Issue resolution workflow documentation connecting PR merges with issue closure.

---

### Exercise 3.3: Open Source Collaboration Workflow Integration
**Objective**: Integrate pull requests into complete open source collaboration and quality assurance workflows.

**Tasks**:
1. Document the complete PR workflow: propose changes → create PR → review → merge → resolve issues
2. Create communication protocols for PR descriptions that help maintainers understand contributions
3. Design quality assurance processes: clear titles/descriptions, conflict checking, maintainer review
4. Establish the role of PRs in building quality through communication, review, and collaborative development

**Deliverable**: Complete open source collaboration workflow document with PR integration and quality assurance protocols.
