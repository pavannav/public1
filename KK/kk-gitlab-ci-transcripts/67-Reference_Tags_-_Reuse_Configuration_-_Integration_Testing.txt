Let's talk about templates and how you can use them within your GitLab CI/CD configuration.
While GitLab CI provides powerful tools for defining your CI/CD pipelines,
manually writing YAML code for every project can be tedious and prone to error. This is where
templates come into the picture. It offers several compelling advantages that enhance your DevOps
workflow. GitLab CI templates are pre-configured YAML files that you can include in your GitLab
CI YAML file to define your pipeline stages and jobs. GitLab already has numerous
pre-made templates for different programming languages, security scanning, and others. So
these templates basically come in two main types. So you have the pipeline templates and the job
templates. Pipeline templates define an entire CI/CD pipeline for a specific project type
or language like NodeJS application or a Ruby on Rails project. And whereas the job templates
define individual pipeline jobs such as a unit testing job, a security job like container scanning,
or a Docker image build job. So by using templates, you can save a lot of time and effort
by not writing the pipeline YAML code for your pipeline from scratch. And you can also easily
reuse templates across different projects. So consider there is a pre-built template tailored
for the NodeJS project that streamlines testing, building, and deployment process.
So it may include stages for unit testing to ensure code quality, code coverage to measure
how thoroughly the tests cover the code, building and pushing Docker images to Docker Hub for easy
sharing and deployment, and finally deploying to Amazon EKS, which is Elastic Kubernetes Service
for scalable container orchestration. So let's understand how this template can be used.
So let's assume we have team A, which is working on a project A. So it has adopted the NodeJS
template. It integrated it into their CI/CD GitLab CI configuration. And they have also adjusted some
test configurations and added credentials for the Docker Hub and the AWS S3. And hence,
they have automated the workflow. So they use the pipeline to efficiently test, build,
and deploy the application to AWS EKS. So here, the team A did not reinvent the wheel by writing
the GitLab CI configuration from scratch. They used an existing template, and they made some
small configurational changes like adding the credentials for their own Docker Hub accounts or
for connecting to their own AWS EKS cluster. And they just used the existing pipeline template.
So this is the way the templates help the teams to efficiently reuse existing pipeline configuration.
So let's take the same template and assume there is a team B working on project B,
and they also want to adopt the same NodeJS template. Let's assume they reuse the template's
core components by leveraging the unit testing and the code coverage stages, but
they have a different containerization and deployment strategy. So what they can do is,
they can modify the other stages and tailor them to push the images to Google Container Registry
and deploy to the Google Kubernetes Engine (GKE) instead of the EKS. So the existing template
can be used as building blocks. So you can use a couple of blocks which are existing,
and you can add your own stages as per your requirement. So what are the key takeaways?
So reusable templates promote efficiency. They save time and effort by providing a pre-built
structure for common tasks. Teams can customize and adapt templates to meet their specific needs
and infrastructure choices. And modular design enables selective reuse. Teams can pick and
choose the stages they need, making the template adaptable to various project requirements.
So how do we store the templates? Where do we access them? So templates may be stored
in various locations including local repositories, private repositories,
GitLab provided templates, and templates from other projects. So how can we reference and
include these templates in our pipelines? So to include an external YAML file into your
CI/CD configuration, we can utilize the includes keyword. So let's take an example. So let's assume
you have a job template within the same project directory as the .gitlab-ci.yml file.
So we can make use of the include local keyword to include a file from the same repository and
branch. We can also use a shorter syntax to specify the path without the local keyword as well.
And for job templates which are hosted in a private GitLab instance project, we can use
the include remote keyword to include the YAML files from another GitLab instance. And similar
to the local keyword, a shorter syntax without the remote keyword can also be used here as well.
However, be careful when you are including another project's CI/CD configuration file
because from a security perspective, this is similar to pulling a third-party dependency.
And if the templates are in another private project of the same GitLab instance,
we can use the include project keyword. And the possible inputs for this type are,
you know, you can include the project name which contains the full GitLab project path,
you can include a file path or an array of file paths relative to the root directory.
The YAML file must have the .yml or .yaml extension. And you can also add an optional
include ref which specifies the ref to retrieve the file from. And if you don't mention this,
it always defaults to the head of the project when not specified.
And one last thing is if you wish to use the GitLab's pre-built templates such as the code
quality template or security scanning templates which are generally stored within the GitLab
official repository under the lib/GitLab/CI/templates directory, you can utilize
the include template keyword to include it within your GitLab CI configs.
Additionally, you can even control when a template is included using rules. For example,
with rules and if conditions, we can include the code quality template only when the commit branch
name starts with a key or a value as feature. Thank you.
