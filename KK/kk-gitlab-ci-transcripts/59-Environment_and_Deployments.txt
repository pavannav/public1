In this session, we will understand what, why, and how about GitLab Kubernetes Agent.
In our previous demo, we deployed an application to Kubernetes by establishing a connection to
the cluster through the Kubeconfig file which is stored as a variable. So your Kubeconfig file
contains sensitive information like the URL of the Kubernetes API server, authentication
credentials, and the default namespace you use. If this information falls into the wrong hands,
they could gain unauthorized access to your Kubernetes cluster and can perform malicious
actions such as deploying malware, deleting resources, or even stealing data. So despite
storing the Kubeconfig file as a masked CI/CD variable within your GitLab server, it still
poses a security concern. Beyond security considerations, there is also a potential
for increased complexity when establishing a connection to the Kubernetes cluster,
especially if the cluster is situated behind a firewall or a network address translation,
which is NAT. So to address these issues, let's explore what the GitLab Agent for Kubernetes
brings to the picture. So the GitLab Kubernetes Agent is a key element within GitLab's
suite of tools designed to facilitate GitOps style deployments. Yes, you heard it right.
So GitLab facilitates GitOps through its integration with Flux. Note that this course
will not explore GitLab's GitOps capabilities, but with GitOps, you can also manage containerized
clusters and applications from a GitLab repository that acts as a single source of truth for your
system. While GitLab Agent's primary role involves GitOps implementation, its functionality extends
far beyond this particular use case. It provides a means of secure communication between GitLab
and the Kubernetes clusters without the need for exposing the Kubernetes API
and storing the credentials in third-party systems. The agent is implemented in a client
server configuration. The Kubernetes cluster side component is called Agent K and the GitLab
server side component is called KAS, which is short for GitLab Agent Server. Communication
between these agent components is initiated by Agent K because the connection must be initiated
from inside the Kubernetes cluster to bypass any firewall or NAT that the cluster may be located
behind. As the channel is bidirectional in nature, in the context of GitOps, this enables either push
or pull-based operations. So here, the GitLab server is the main application which talks to KAS.
So AgentK is the GitLab agent. It's going to keep a connection established to the KAS
instance, waiting for a request to process. It may also actively send information about things
which are happening within the cluster. But what about KAS? KAS is responsible for accepting
requests from the AgentK, authentication of requests from AgentK by querying GitLab.
After that, it's also going to fetch the agent's configuration from a corresponding Git repository
by querying Gitly. So what is Gitly? So Gitly provides high-level RPC access to Git repositories
and it is used by GitLab to read and write Git data. This is the method through which the AgentK
establishes communication, forwarding requests, responses, sending notifications, and also handles
other interactions between them. GitLab server takes care of managing the KAS
and it's going to eliminate the need for users to upgrade AgentK within their clusters whenever
there are updates or additional features. And GitLab also prefers implementing modifications
and logic directly to KAS instead of the AgentK. And given the lightweight nature of AgentK,
GitLab suggests having at least one instance of AgentK per cluster.
So how do we register this? So registering a GitLab Kubernetes Agent, you know, it's going to
connect with your Kubernetes cluster to GitLab. So what are the prerequisites? So we need to have
a GitLab account with sufficient permissions to create and manage projects and the Kubernetes
cluster. Whereas the GitLab project is where we want to, you know, register the agent and this
project will host the agent configuration files and it's also going to manage the deployments.
And finally, we also need a Kubernetes cluster which meets the minimum version requirement for
the GitLab agent compatibility. So to register, we have to navigate to the GitLab project,
click on the Operate and then choose Kubernetes cluster. We can create a new agent by specifying
a unique name. GitLab is going to automatically generate a unique and secure token for the agent
and it's also going to, you know, give you the Helm installation commands. The Helm chart
uses the generated token and the KAS address to enable the registration and the communication
process. And by running the Helm chart commands on your Kubernetes cluster, we can complete the
installation phase. After successful installation, the agent connects to the GitLab project or the
server and becomes visible in the Kubernetes cluster section of your project. We can further
manage the GitLab agent in your project using a config file located at the .gitlab
slash agent slash the agent name slash config.yaml file within your project repository.
This file is going to allow you to specify GitHub settings which define how the agent monitors
changes and triggers deployments for automated workflows. It will also help you to set CI/CD
access control to limit which projects and users can use the agent for deployments. And finally,
you can also configure remote development by defining settings
for creating and managing workspaces within your clusters. Thank you.
